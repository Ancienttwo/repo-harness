**裁决：有条件通过，暂不建议按原稿直接执行。**

# 一、Program 级 P0：VGBR 不能只用一个普通 worktree

这是当前方案中最重要的执行矛盾。

VGBR runner 的 authoritative 模式要求：

* source checkout 必须干净；
* `source_commit` 取运行目录当前 `HEAD`；
* subject hash 包含 runner、manifest、fixtures、`package.json`、`src/cli`、`assets` 等内容；
* validator 会针对当前 source checkout 重新计算 subject 并验证报告；
* authoritative report 必须是完整、唯一的 3×9 matrix。([GitHub][3])

但拟稿同时要求：

```text
VGBR plan / contract / review / notes
+ report files
+ HEAD == POST_HRD_SHA
+ checkout clean
```

如果这些文件在同一个 VGBR branch/worktree 里：

* 未提交时，checkout 不干净；
* 提交后，`HEAD` 已经不再等于 `POST_HRD_SHA`；
* 报告记录的 `source_commit` 会变成 VGBR planning commit，而不是待测基线；
* 因此 contract 无法同时满足全部条件。

## 必须改成双 checkout 模式

### A. Orchestration worktree

拥有：

```text
plan
contract
review
notes
Program annotation
最终 report artifacts
```

可以是：

```text
codex/vgbr-post-hrd-baseline-recovery
```

### B. Detached subject checkout

特点：

```text
detached HEAD @ exact POST_HRD_SHA
clean
read-only subject
不投影 active-plan
不创建 task workflow 文件
不提交任何内容
```

运行流程应改成：

```text
VGBR orchestration WT
        │
        ├── preflight / contract / attempt record
        │
        └── detached subject checkout @ POST_HRD_SHA
                  │
                  ├── authoritative benchmark exactly once
                  ├── validate report against same subject
                  └── report written to external staging directory
                                │
                                ▼
                 copy verified artifacts into orchestration WT
```

建议 contract 明确区分：

```text
CONTROL_REPO
SUBJECT_REPO
EXPECTED_BASE_SHA
EXPECTED_SUBJECT_SHA
REPORT_STAGE_DIR
ATTEMPT_ID
```

主体命令可以保持不变，但 `--report` 应指向 subject checkout 外部的 staging path：

```bash
bun scripts/run-harness-profile-benchmark.ts \
  --require-authoritative \
  --provider codex \
  --manifest evals/harness/scenarios.json \
  --report "$REPORT_STAGE_DIR/profile-comparison.json"
```

随后在同一个 detached subject checkout 中验证：

```bash
bun scripts/validate-harness-profile-benchmark.ts \
  --report "$REPORT_STAGE_DIR/profile-comparison.json" \
  --require-authoritative \
  --format json
```

通过后才复制到 orchestration WT 的 canonical report paths。

# 二、Program 级 P0：SHA 命名存在自指问题

当前文本说 `HRD-CLOSEOUT` 要“记录最终 post-HRD SHA”。但一个 PR 在合并前，不可能在自己的文件里预先记录自己的最终 merge SHA。

同样的问题也会发生在：

* VGBR-R 试图记录自身 merge 后的 `POST_VGBR_SHA`；
* EPC-09 试图记录自身 merge 后的 `POST_EPC_SHA`。

这会产生无穷的“再开一个提交补上前一个提交 SHA”。

## 建议采用 successor-pinned 规则

定义：

```text
HRD_RUNTIME_SHA
  = b5a98c903d3728002d2f663ba7a1b421913e368f
  = PR #106 merge commit
```

然后：

```text
POST_HRD_SHA
  = HRD-CLOSEOUT merge 后，由 VGBR-R contract 在 fresh fetch 后钉住
```

同理：

```text
POST_VGBR_SHA
  = VGBR-R merge 后，由 EPC-00 contract 钉住

POST_EPC_SHA
  = EPC-09 merge 后，由 SSD activation contract 钉住
```

因此，`HRD-CLOSEOUT` 中只应记录：

```text
HRD_RUNTIME_SHA: b5a98c9...
Closeout subject: current lifecycle projections
Successor pin: required
```

而不是尝试写入尚不存在的自身 merge SHA。

## 建议增加统一规则

```markdown
### Successor-pinned SHA rule

A package records the exact base SHA it consumed.
It never predicts its own merge SHA.
The first dependent successor fetches origin/main after merge,
pins that exact SHA, and records it as POST_<PREDECESSOR>_SHA.
```

这是整个 Program 应统一采用的规则。

# 三、Program 级 P0：BDD2 虽然是并行 Program，但会改变 VGBR subject

“BDD2 不共享 contract、branch、PR、写路径”还不足以证明它可以和 VGBR 并行。

你上传的 BDD2 follow-through plan 明确会修改：

```text
src/cli/hook/prompt-intents.ts
src/cli/commands/prompt-guard-decision.ts
assets/hooks/prompt-guard.sh
assets/templates/...
assets/reference-configs/...
```

也就是 `src/cli` 和 `assets`。

而 VGBR benchmark subject 正好包含：

```text
package.json
src/cli
assets
runner
manifest
fixtures
```

因此，BDD2 即使 allowed paths 与 report paths 完全不重叠，也会改变 benchmark 的待测对象。([GitHub][3])

## 必须把 concurrency gate 从“写路径不重叠”提升为两层

### 第一层：repository ownership overlap

```text
changed paths ∩ other package allowed_paths == ∅
```

### 第二层：verification subject overlap

```text
other package changed paths ∩ benchmark subject inputs == ∅
```

对 VGBR 而言，第二层更重要。

## 推荐执行策略

在 VGBR 的 code freeze 到报告合并期间：

```text
禁止任何会修改 package.json / src/cli / assets /
benchmark runner / scenario manifest / fixtures 的 PR 合并到 main
```

BDD2 worktree 可以继续开发，但不能在 VGBR subject freeze 后合并。

更严格的流程是：

```text
1. 决定 BDD2 是否进入本次 baseline
2. fresh fetch origin/main
3. pin POST_HRD_SHA / actual baseline SHA
4. 宣告 subject quiescence
5. 执行 VGBR
6. 合并 VGBR report
7. 解除 subject quiescence
```

如果 BDD2 在 VGBR 前已经合并，就不要把报告描述成“纯 post-HRD baseline”，而应写成：

```text
current pre-EPC baseline after HRD and acknowledged parallel changes
```

不应在命名上掩盖实际 subject 内容。

# 四、Program 级 P0：VGBR profile 名称和能力声明不准确

当前 runner 的三个 profile base 是：

```text
no-harness
adaptive-lite
strict-harness
```

其中 `adaptive-lite` 使用的是 standard deployment profile；当前 matrix 并不存在彼此独立的 `Lite` 和 `Standard` 两个 arm。完整 authoritative matrix 是上述三个 base × 九个场景，共 27 arms。([GitHub][4])

Program audit 本身也已经指出这一盲点：当前 arm 把 adaptive-lite 与 standard deployment 混在一起，未来若要明确分辨 no-harness / lite / standard / strict，应当是四个 arm，而不是现有三 arm。([GitHub][5])

因此，VGBR-R contract 不能写：

```text
Lite / Standard / Strict 三个 profile base
```

应改为：

```text
no-harness / adaptive-lite / strict-harness
× 9 scenarios
= 27 authoritative arms
```

其结论也只能是：

> **恢复当前 runner 所定义的 authoritative 3×9 baseline。**

不能声称：

> 已独立验证 Lite、Standard、Strict 三档语义。

要做四档比较，就必须修改 runner/manifest，因而不再是 eval-only VGBR-R，而是另一个获批的 benchmark-design package。

# 五、VGBR 一次性运行策略需要再精确一层

“冻结后仅一次，失败不自动重跑”是正确方向，但需要区分：

```text
一次 invocation
一次 valid attempt
一次 accepted report
```

runner 能锁住并发昂贵运行，但未必能证明历史上总共只被调用了一次。因此“invocation count == 1”不能只写成最终报告断言，它需要外部 attempt evidence。

## 建议增加 attempt contract

在运行前生成：

```text
ATTEMPT_ID
EXPECTED_SUBJECT_SHA
EXPECTED_SUBJECT_HASH
started_at
command_sha256
provider
provider_cli_version
```

存放在 orchestration WT 的 run evidence 中，而不是 detached subject checkout。

执行结果只能是：

```text
accepted
failed_before_provider
failed_during_run
invalid_report
cancelled
```

任何非 `accepted` 结果：

```text
不得自动再次运行
不得使用 --regrade-existing
不得缩小 profile/scenario 后补跑
必须重新形成 run-decision / waiver / 新 contract
```

特别应明确禁止：

```text
--profile
--scenario
--regrade-existing
```

## 25–40 分钟的正确表达

可以保留：

```text
Expected operator budget: 25–40 minutes
```

但不能作为 pass/fail 条件。runner 自己的 hard-wall budget 是 50 分钟；超过你估计的 40 分钟但低于 runner hard limit，不应自动判为无效。([GitHub][3])

## Machine acceptance 应预先冻结

在启动 benchmark 前，应把下列 rubric 写死：

* `authoritative == true`
* `source_commit == EXPECTED_SUBJECT_SHA`
* subject hash 与当前 detached checkout 一致
* profile bases 恰好为三个预期值
* scenario 数恰好为 9
* records 恰好为 27
* 每个 `(profile_base, scenario_id)` 唯一
* workspace 与 provider home 隔离且唯一
* structured provider/grader records 完整
* report byte binding 有效
* validator 通过
* matrix test 通过
* 无 subject drift
* 无第二次执行

旧报告确实来自 `606b02c…`，而当前 runner 会把 `src/cli`、`assets` 等纳入 subject；因此它不能作为当前 post-HRD 的精确基线，这是由 subject 变化推导出的结论。([GitHub][6])

# 七、EPC-00 目前仍不足以作为 EPC-01 的启动授权

把 EPC-01～09 从 audit backlog 变成 canonical Sprint 是必要的，但 EPC-00 不能只做“文档搬运”。

当前 Program audit已经指出：

* `checks/latest`、handoff、resume、current 等存在 authority/projection 混合；
* EPC 必须决定哪些 view 保留、哪些退休；
* 旧 writer 应在 authority cutover 的同一 package 中删除；
* Context Packet 有明确的 token 上限与 p95 目标。([GitHub][5])

因此 EPC-00 必须冻结以下九类设计决策，否则 EPC-01 会在实现中临时发明 authority semantics。

## 1. Ledger topology

至少明确：

```text
store path
per-worktree or shared
tracked or gitignored
retention
compaction
branch/reset behavior
cleanup ownership
```

我的建议是：

```text
.ai/harness/evidence/events/   per-worktree, gitignored
.ai/harness/evidence/blobs/    content-addressed, gitignored
tracked review/receipt         only records accepted event IDs and hashes
```

不要让多个 worktree 共享一个可变 event store。

## 2. Cutover epoch

不应追溯伪造历史 event。

建议：

```text
ledger_epoch_start_sha = exact EPC-01 cutover base
pre-epoch artifacts = legacy evidence, readable but not backfilled
post-epoch evidence = EvidenceEvent only
```

## 3. Subject identity

不能只用 `HEAD SHA`。至少要能表达：

```text
authority commit
base commit
target commit
scope hash
subject hash
contract hash
command hash
environment/provider identity
```

## 4. Trust matrix

建议 EPC-00 先冻结：

| Evidence source              |                默认 trust | 能否满足 machine gate |
| ---------------------------- | ----------------------: | ----------------: |
| authoritative verify runner  | `authoritative_machine` | 仅 exact subject 可 |
| PostBash observation         |              `observed` |                不可 |
| human receipt                |      `human_acceptance` |       只能满足人工 gate |
| external importer            |     `external_attested` |          仅合同显式允许时 |
| projection/materialized view |               无独立 trust |     永远不可自行满足 gate |

`checks/latest`、handoff 或 Markdown projection 不能重新升级成 authority。

## 5. Event identity 与 replay

需要冻结：

```text
event_id generation
idempotency key
ordering
supersedes relation
duplicate import behavior
replay determinism
corrupt tail recovery
```

## 6. Blob 与安全边界

需要明确：

```text
最大 payload
secret redaction
absolute-path prohibition
symlink handling
binary evidence
hash algorithm
content-addressed blob ownership
```

## 7. `checks/latest` 选择规则

不能再是：

```text
mtime 最新
文件名最新
最后写入者获胜
```

必须是：

```text
当前 worktree
+ 当前 active contract
+ exact subject
+ accepted trust class
+ deterministic event order
```

## 8. Projection provenance

每个 projection 应携带：

```text
schema_version
generated_at
source_event_ids
source_checkpoint_id
subject_hash
materializer_version
```

从而可以证明 projection drift。

## 9. Recovery view 最小化决策

不要在 EPC-07 中预先假定以下四个都必须长期存在：

```text
handoff
resume
tasks/current
task-handoff
```

EPC-00/EPC-07 应先做 consumer inventory，然后决定：

```text
keep
merge
retire
```

Program audit明确要求 EPC 决定哪些 view 继续 canonical、哪些退休，而不是把现有所有 view 都重新实现一遍。([GitHub][5])

# 八、EPC-06 和 EPC-07 的措辞需要修正

## EPC-06：checkpoint 不是新的权威

当前：

> 一个 checkpoint transaction 同时生成 machine/human view。

建议改为：

> 一个 checkpoint materialization transaction，从同一组 accepted EvidenceEvents 生成 canonical machine projection，并由该 machine projection 确定性地生成 human-readable view。

权威关系应是：

```text
EvidenceEvent ledger
        ↓
checkpoint JSON projection
        ↓
checkpoint Markdown projection
```

而不是：

```text
JSON authority ↔ Markdown authority
```

Markdown 不能成为第二套可人工改写的 authority。

另外，不要承诺跨任意多个文件的“绝对原子性”。更现实的合同是：

```text
stage all outputs
validate source event set and hashes
rename/install batch
publish checkpoint marker last
detect and reject partial generation
```

## EPC-07：先删重复 surface，再统一 writer

建议改名为：

```text
EPC-07 consumer inventory + minimal recovery projection cutover
```

顺序：

```text
1. 枚举所有 reader/consumer
2. 决定每个 view keep/merge/retire
3. 为 surviving views 建唯一 materializer
4. 在同包删除 retired writers
5. 添加 no-independent-authoring test
6. 添加 projection drift test
```

不能把“所有现有 view 都从 ledger 重建”本身当作目标，否则会把历史重复结构永久化。

# 九、EPC-02/03/04：默认串行，并行只是资格

你现在的并行条件是合理的，但还应更严格：

```text
EPC-01 merge
+ schema frozen
+ trust matrix frozen
+ idempotency frozen
+ exact allowed_paths disjoint
+ test fixtures disjoint
+ no shared barrel/export file
+ no shared store writer
+ no shared projection writer
```

实际实现中，EPC-02、03、04 很可能都会触碰：

* EvidenceEvent types；
* importer registry；
* fixtures；
* exports；
* common validation；
* trust enum；
* store append API。

因此建议 Program 写成：

> **串行为默认；并行必须由三个 contract 联合证明资格。**

而不是：

> 默认 optional parallel，发现冲突后再退回串行。

这样更符合 fail-closed。

# 十、EPC-09 范围过宽，需要防止“垃圾都留到最后清”

建议 EPC-09 只保留：

```text
cross-package projection drift check
final deprecation residue scan
post-EPC evaluation
release notes
Program closeout
successor handoff to SSD
```

## 还需要明确 VGBR 的后测用途

若 VGBR 是 EPC 前 baseline，那么 EPC-09 应二选一：

### 方案 A：执行严格可比的 post-EPC benchmark

* 同一 runner/manifest；
* 同一三 profile base；

* 同一 acceptance rubric；
* 冻结 post-EPC subject；
* 运行一次；
* 报告 before/after 差异。

### 方案 B：不执行匹配后测

那就必须把 VGBR 标记为：

```text
descriptive pre-EPC baseline only
```

不能在 release closeout 中声称 EPC 改善了 benchmark 指标。

另外，当前 runner 没有显式固定 Codex 模型，只记录 CLI/version 等运行环境。因此除非 EPC-09 对 provider 配置做同等冻结，否则性能差异应视为描述性证据，而不是强因果证据。([GitHub][3])

# 最终审批语句

建议给 Fable 的裁决改成：

> **APPROVE WITH REQUIRED AMENDMENTS。**
> 接受 post-HRD VGBR recovery、独立 HRD closeout、一次性 authoritative baseline、EPC package 化和 SSD no-touch 的总体方向。
> 在开始执行前，必须修复 SHA 自指、VGBR 单 worktree 矛盾、profile 命名错误和 benchmark subject concurrency 缺口。
