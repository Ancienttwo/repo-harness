# 结论

我审阅了当前 `main` 中的 `evals/bdd2`、评测 runner、冻结报告、truth/tasks/rubrics、后续新增的 `design-options`、`ux-feature-guard`、`repo-harness-prd`、external tooling 约定和相关测试。以下是静态 code review；我没有把仓库测试包装成“已在本地重新跑过”的结果。

**直接回答你的两个问题：**

1. **BDD² 单独不能实现你的需求。**
   它是一套设计严谨的离线评测与研究框架，能识别 coding agent 的无依据 UI 扩张，但不是当前 repo-harness 运行时真正执行的 guard。更关键的是，它自己的 E3 结果并未证明现有干预有效。

2. **当前 `main` 已经用 `design-options + UX Feature Guard` 吸收了 BDD² 的主要教训，方向是对的，但还只能算“软约束”。**
   它能显著改善 agent 的决策方式，却不能保证最终实现里一定没有 developer-view leakage。

3. **外部 skill 建议独立，但不要做成通用的 `external-skills` 插件框架。**
   更合适的是一个独立的 **`preview/design-decision` 工作流模块**：它拥有状态、权限边界、用户选择与 handoff；peer research、imagegen、taste 只是可替换的 host capability/provider。

一句话架构判断：

> **保留 BDD² 作为评测实验室；把 design-options 升级为 preview 工作流；给 UX Feature Guard 补一个按 feature 定义的 rendered-surface gate；不要再造通用 adapter/ledger/catalog。**

# 一、BDD² 做对了什么

## 1. 它非常准确地定义了你真正反感的 failure mode

BDD² 不只是笼统地说“不要 over-engineering”，而是检查：

* 未获授权的用户可见 choice、concept、state、role、interruption；
* backstage leakage；
* journey fragmentation；
* 缺失 recovery；
* authority conflict；
* 为未来假想需求提前增加设置、dropdown、toolbar、batch action。

任务 truth 甚至明确把 `Model / Prompt / Queue / Fallback settings` 定义为未经授权的用户概念，并要求不得暴露内部 orchestration controls。这和你描述的“coding agent 总喜欢画蛇添足做 developer view”几乎完全对齐。 ([GitHub][1])

因此，**BDD² 的问题定义是强的，应当保留。**

## 2. Browser 和 ImageGen 的 authority ceiling 定义得很好

BDD² 已经意识到：

* peer research 可以证明某种模式真实存在；
* 可以帮助判断 visual structure 和 candidate fit；
* 但不能证明产品需要这个功能；
* 不能从案例推导 retry 次数、threshold、policy；
* imagegen 只是 synthetic stimulus，不能证明用户偏好、需求、信任、可用性或 accessibility。

这套边界后来被完整带入 `design-options`：browser 是普通 host call，不是权威 adapter；imagegen 最多生成 2–3 个刺激图；最后必须由用户关闭 preference decision。 ([GitHub][2])

这也是未来接入 taste skill 时必须延续的原则：**skill 可以提供证据、刺激和 refinement，但不能拥有产品语义。**

## 3. 评测工程严谨

BDD² E3 有固定 manifest、hash 校验、隔离 workspace、双 reviewer、adjudicator、冻结证据、指定 CLI/model/version 和严格 schema。Implementation pilot 也被设计为只有 shape 和至少一个 adapter 通过后才能运行。 ([GitHub][3])

所以它很适合作为：

* regression benchmark；
* prompt/intervention 研究环境；
* 新 preview workflow 的离线对照组；
* 防止未来重新引入已经失败的重型机制。

# 二、为什么 BDD² 当前仍然不能满足需求

## 1. 三个核心 E3 实验全部是 Kill

BDD² 的 terminal results 是：

* `S3`: Kill；
* `EB3`: Kill；
* `EI3`: Kill；
* `I3`: 因前置 gate 未通过而没有运行。

也就是说，当前证据并没有证明 inline Behavior Card、Browser Evidence Adapter 或 ImageGen Prototype Adapter 能稳定降低 unsupported UI expansion。Implementation pilot 没有形成可用的端到端证据。 ([GitHub][4])

这不代表 BDD² 没价值，而是它证明了：

> **问题真实存在，但原来的产品化形态没有通过自己的标准。**

## 2. BDD² 是 eval，不是 runtime module

当前测试明确断言 Behavior Audit 和 Phase P 产品 surface 不存在，并检查以下路径不存在：

* `assets/skill-commands/repo-harness-bdd`
* `src/cli/commands/bdd.ts`
* `src/cli/mcp/behavior-tools.ts`

同时，S3、EB3、EI3 的 Kill 结果被作为冻结历史证据保留。 ([GitHub][5])

因此，BDD² 现在不会在 coding agent 实现 frontend feature 时自动：

* 阻止额外 route；
* 阻止 model/provider selector；
* 检查最终 DOM；
* 检查可见文本；
* 阻止 taste skill 引入额外状态；
* 拦截不在 PRD 中的 developer diagnostics。

它能评估，却不能直接执行保护。

## 3. 它没有评估真实的外部 skill orchestration

E3 manifest 中明确关闭了 browser、web search、MCP、external tools 和 repository tools；使用的是预先准备好的 evidence，而不是让 agent 真正执行 peer research、imagegen 或 taste workflow。 ([GitHub][6])

因此它回答的是：

> “给 agent 一份这种 evidence，会发生什么？”

而不是：

> “agent 在真实任务里能否正确判断何时研究、何时生成图、何时停止、何时请求用户、何时调用 taste？”

后者正是你现在真正需要补的模块。

# 三、当前 `main` 已经比 BDD² 前进了一大步

## 1. `design-options` 是正确的 BDD² 后续形态

它的规则很清楚：

* 只有真正存在 2–3 个方向时才触发；
* agent 收集 reference evidence；
* 用 imagegen 生成 2–3 个 preview；
* agent 不得自己选择；
* agent 不得推荐某一个；
* 用户选择是唯一能关闭 decision 的事件；
* 用户不在场时，展示 options 后停止。

它还明确说明这不是复活 BDD² 的 adapters，而是把 browser/imagegen 降为普通 host calls，并刻意不引入 ledger、lifecycle、sidecar、catalog。 ([GitHub][7])

这正是应该保留的设计。

## 2. UX Feature Guard 建立了正确的 authority chain

当前 guard 已要求：

* 实现前冻结 user-visible behavior；
* copy、interaction、persistence、error state、component reuse 都属于产品行为，不能在实现中临时发明；
* 有视觉分歧时先跑 design-options；
* 用户选择是 authority；
* design brief 填写 positive、negative/non-goal、authority-failure 场景；
* frontend Sprint 或 contract 前必须获得明确的人类确认。 ([GitHub][8])

它还建立了合理的单向权威链：

```text
用户决定 / 产品文档 / 已批准 PRD
                ↓
       confirmed design brief
                ↓
          task contract
                ↓
             BDD tests
                ↓
         implementation evidence
```

BDD test 是 projection，不是新的产品 authority；冲突时应回到最早的权威层，而不是让实现或测试偷偷覆盖产品语义。 ([GitHub][8])

## 3. PRD skill 已经接上了这条链路

`repo-harness-prd` 当前已经：

* 在 UI/taste、market pattern、framework choice 等情况下触发 prior-art research；
* 每个 P0 module 必须有 negative/non-goal scenario；
* frontend PRD 必须先读取 UX Feature Guard；
* 必须生成 design brief；
* 必须填写稳定的正向、负向和 authority-failure scenario ID；
* `imagegen-frontend-web` 和 `design-taste-frontend` 被定义为 optional preview enhancer，不能替代人工 checklist。 ([GitHub][9])

所以你设想的工作流并不是完全不存在，而是目前散落在：

* PRD skill；
* design-options convention；
* UX Feature Guard；
* design brief；
* host external skills。

缺的是一个明确的 orchestration owner。

# 四、当前实现的关键缺口

## 1. 它仍然是 soft gate，不是 executable guard

UX Feature Guard 明确选择了“不增加 validator、ledger、sidecar、catalog、scoring model 或 semantic classifier”。repo-harness 的 prompt hooks 即使处于 `enforce` 配置，也会归一化为 advisory；真正的 enforcement 仍留给测试、contract 和 human review。 ([GitHub][8])

这避免了过度工程化，但带来一个直接结果：

> Agent 可以完整填写 design brief，然后仍然在代码里加入一个 “Advanced model settings” panel。

当前没有确定性的 rendered-surface gate 阻止它。

## 2. 当前测试主要证明“约定存在”，不证明“最终 UI 合规”

`ux-feature-guardrail.test.ts` 当前验证的是：

* docs mirror 一致；
* template 中包含指定字段；
* `docs show` 能读取 guard；
* PRD 和 prompt hook 指向 guard；
* adoption 会把文件投影到下游 repo；
* 没有新增 BDD ledger。

这些是重要的 packaging tests，但没有生成页面、检查 DOM、检查 accessibility tree，也没有判断最终用户可见 concept 是否超出 brief。 ([GitHub][10])

## 3. Developer view 缺少 role-aware contract

当前已经有 `Forbidden extras`、`Observable & Copy Contract` 和“required presence/absence”，但还缺少显式的：

* 这个 surface 面向谁；
* 哪些 concept 对这个 persona 合法；
* 哪些内部 concept 必须保持 backstage；
* developer/admin 角色是否存在例外；
* diagnostics 是否属于用户请求。

这不能做成全局词表，因为对开发者工具而言，`model`、`prompt`、`trace ID` 有时就是产品功能。

正确判定应该是：

> **某个可见 concept 是否由当前用户角色、请求结果、下一步行动或 recovery 所需要，而不是这个词本身是否“技术化”。**

## 4. Taste 只有名称，没有正式 authority ceiling

当前 PRD skill 仅把 `design-taste-frontend` 称作 optional enhancer，主路由表没有 taste 的独立执行顺序或输出约束。 ([GitHub][11])

这会留下一个危险空间：taste skill 可能为了“更完整”而加入：

* advanced settings；
* provider selector；
* debug status；
* queue/history；
* dashboard metrics；
* extra empty/loading states；
* 不必要的 navigation。

Taste 的合法权限必须被限制为：

```text
可以：
视觉层级、spacing、typography、tokens、component selection、
已有 interaction 的表现形式、已冻结 copy 的表达质量

不可以：
新增 feature、state、role、route、field、setting、product policy、
persistence semantics、retry rule、diagnostic mode
```

它想做语义变化时，只能产生 proposal，不能直接写进设计或实现。

## 5. `design_options_choice` 存在 policy/schema 漂移

新增 design-options 路由时，commit 明确把 `design_options_choice` 加进了 `.ai/harness/policy.json`，但没有更新 `workflow-contract`，原因是后者仍是 closed six-field TypeScript interface，当时被列为 out of scope。 ([GitHub][12])

这意味着当前它更像 prompt/convention routing，而不是完整的 typed workflow contract。正式增加 preview 模块时，这个漂移应该被解决。

# 五、推荐的 Preview 工作流顺序

你提到的：

```text
peer research
→ imagegen
→ PRD
→ taste
```

总体方向是对的，但我建议调整一个关键点：

> **peer research 可以在完整 PRD 前发生；imagegen 则至少要在一个轻量 product boundary 冻结之后发生。**

否则 imagegen 很容易把未请求的功能视觉化，而 agent 又会反过来把图中的内容写入 PRD。

推荐完整链路：

```text
1. Problem Boundary
   目标用户 / 用户结果 / 当前行为 / 非目标 / 禁止扩张
                         ↓
2. Peer Research
   只研究 pattern、structure、candidate fit
                         ↓
3. Draft PRD
   冻结 feature need、product policy、role、non-goals
                         ↓
4. Design Options
   若确有多个视觉方向，imagegen 生成 2–3 个 STIMULUS
                         ↓
5. Human Choice
   用户选择写入 user_evidence
                         ↓
6. Taste Refinement
   只优化被选方向，不得新增产品语义
                         ↓
7. Confirmed Design Brief
   observable contract + concept contract + BDD scenarios
                         ↓
8. Contract / Implementation
                         ↓
9. Rendered-Surface Verification
   检查最终用户真正看到的内容
```

这里要区分两个 PRD 层次：

* **Draft PRD/product boundary**：必须在 imagegen 前限制“这个功能是什么、不是什么”；
* **Approved PRD + design brief**：可在用户选择和 taste refinement 后完成最终确认。

这样既允许案例研究和 preview 参与产品设计，又不会让 synthetic image 成为 feature authority。现有 design-options 已明确 browser evidence 不能证明 feature need/product policy，imagegen 也只能是 STIMULUS。 ([GitHub][7])

# 六、外部 skill 是否做独立模块

## 我的建议：做独立模块，但模块名和职责应是 `preview`，不是 `external-skills`

“外部”只是部署和调用方式，不是产品语义。

建议拆成三层：

```text
┌──────────────────────────────────────────────┐
│ Preview Workflow Core                        │
│ decision state / authority / stop rules      │
│ user_evidence / design-brief handoff         │
└──────────────────────────────────────────────┘
              │            │            │
              ▼            ▼            ▼
       peer_research    image_generation    ui_taste
       host/browser     host/imagegen       external skill
              │            │            │
              └────── untrusted evidence ───┘
```

### Preview Core 拥有什么

它只拥有：

* 是否应该触发 preview；
* 当前开放的设计问题；
* frozen product boundary；
* provider authority ceilings；
* 最多几个方向；
* 是否已展示；
* 是否获得用户选择；
* 是否完成 taste refinement；
* 是否可进入 design brief confirmation。

建议状态机：

```text
boundary_frozen
→ research_complete
→ options_presented
→ user_selected
→ taste_refined
→ brief_confirmed
```

其中只有真实的人类事件才能创建 `user_selected`。

### Provider 拥有什么

建议只定义三个 capability：

| Capability         | 可以输出                                    | 不得输出                                           |
| ------------------ | --------------------------------------- | ---------------------------------------------- |
| `peer_research`    | 案例、pattern、visual structure、适配性         | feature need、policy、threshold、accessibility 结论 |
| `image_generation` | 2–3 个有明确 trade-off 的刺激图                 | 产品批准、默认推荐、隐藏选定方向                               |
| `ui_taste`         | 视觉 refinement、hierarchy、tokens、reuse 建议 | 新 feature、state、role、route、setting、数据字段        |

不要让 provider 直接写 PRD、contract 或 implementation authority。

## 不建议现在做的东西

暂时不要增加：

* 通用 skill catalog；
* provider lifecycle ledger；
* 一套新的 evidence packet schema；
* provider scoring system；
* arbitrary skill chaining；
* 独立 MCP sidecar；
* 一个声称能自动理解所有 UI 语义的 classifier。

BDD² 和后续实验已经提供了足够证据，说明这类 machinery 很容易比原问题更复杂。当前 design-options 也正是有意避开这些机制。 ([GitHub][7])

Repo 当前 external tooling 也倾向于让外部工具运行在自己的 trust boundary，只把 evidence manifest/artifact 引入现有 review flow；自动 provider invocation 目前并不存在。 ([GitHub][13])

# 七、建议增加的 Developer-View Guard

## 1. 在 design brief 增加 role-aware concept contract

建议在现有 UX Feature Guard 中增加：

```markdown
### User-visible Concept Contract

- Audience / role:
  - end_user

- Allowed visible concepts:
  - task
  - progress
  - result
  - failure
  - retry action

- Required recovery concepts:
  - what failed
  - what the user can do next

- Forbidden backstage concepts:
  - model
  - provider
  - prompt
  - queue
  - fallback strategy
  - token usage
  - raw JSON
  - internal trace ID
  - orchestration retry policy

- Role-gated exceptions:
  - none
```

这些只是 end-user 产品的示例，不能成为所有 repo 的全局 denylist。

每个 forbidden concept 都应来自：

* PRD non-goal；
* 当前 persona；
* 未授权的 product concept；
* 明确的 backstage authority。

## 2. 增加明确的 negative scenario

```gherkin
@UX-UPLOAD-N1
Scenario: internal orchestration remains backstage
  Given the current surface is for an end user
  And the PRD does not authorize a diagnostic or developer mode
  When the upload operation is loading, completed, or failed
  Then the user sees only the outcome, current status, and available next action
  And no model, provider, prompt, queue, fallback, token, or raw trace information is visible
  And no additional settings or navigation are introduced
```

同时保留 failure scenario：

```gherkin
@UX-UPLOAD-F1
Scenario: a provider failure is translated into a user recovery path
  Given the authoritative operation fails
  When the result is rendered
  Then the interface states what failed and what the user can do next
  And it does not report success
  And it does not expose the raw provider response as normal end-user copy
```

## 3. 增加 post-render executable verification

不要重新做全局 semantic classifier。应当根据每个 design brief 生成明确、可执行的断言：

* 检查 DOM visible text；
* 检查 accessibility tree；
* 检查可见 controls、links、routes；
* 检查不同 role 的 surface；
* 检查 forbidden selector/test ID；
* 检查 required recovery action；
* 截图作为辅助 review evidence，而不是唯一判断。

例如提供一个小型 helper：

```ts
await assertUxSurface(page, {
  role: "end_user",
  requiredText: ["Upload failed", "Try again"],
  forbiddenText: [
    /model/i,
    /provider/i,
    /prompt/i,
    /queue/i,
    /fallback/i,
    /raw json/i,
  ],
  forbiddenSelectors: [
    "[data-testid='model-selector']",
    "[data-testid='debug-panel']",
  ],
});
```

关键是这些字段来自 design brief，而不是 repo-harness 自己猜测。

## 4. 不要为了反对 developer view 而删除必要的 recovery

BDD² 本身也检查 required omission 和 protected-concern regressions。不能因为“不要技术信息”就：

* 隐藏错误；
* 删除 retry；
* 删除必要的状态；
* 破坏 accessibility；
* 吞掉安全提示。

目标不是“界面越少越好”，而是：

> **每个用户可见元素都能映射到用户目标、当前状态、下一步行动或必要 recovery。**

# 九、BDD² 自身的工程性改进

这些不是 UX 产品 blocker，但值得清理：

1. `scripts/run-bdd2-evals.ts` 当前约 1,300 多行、170 KB，同时承载大量 experiment/schema/runner 逻辑，建议拆成 runner core、experiment definitions、evidence verification 和 report projection。 ([GitHub][14])


* baseline agent；
* preview workflow；
* peer research；
* imagegen；
* taste adversarial injection；
* provider unavailable；
* user absent；
* end-user 与 developer-role 两种 surface；
* 实现后的 DOM/accessibility verification；
* protected concerns 不回退。

| Peer research authority     | 当前 design-options 定义合理             | 可用              |
| Imagegen preview            | 2–3 个刺激图、用户关闭选择                    | 可用              |
| Taste skill                 | 仅为 optional enhancer               | 需要正式边界          |
| PRD/design brief authority  | 基本清晰                               | 需要明确 preview 顺序 |
| 阻止最终 UI 额外内容                | 主要依靠 agent judgment 和 human review | 不足              |
| Rendered-surface gate       | 缺失                                 | 必须补             |
| 通用 external skill framework | 未实现                                | 也不建议实现          |
| 独立 preview workflow module  | 尚未正式存在                             | 建议实现            |

**最终推荐是：**

```text
BDD²               = frozen evaluation laboratory
UX Feature Guard   = product-behavior authority
Design Options     = human-choice protocol
repo-harness-preview = external skill orchestration owner
Rendered Surface Gate = executable anti-developer-view proof
```

这套拆分既能保住 BDD² 已经验证过的 authority 思想，也不会重新走回 adapter、ledger、validator machinery 过重的老路。
