# Transactional Adoption Planner

> **Status**: Canonical apply authority
> **CLI**: `repo-harness adopt [--dry-run] [--mode minimal|standard|self-host]`
> **Planner**: `src/core/adoption/standard-plan.ts`
> **Executor**: `src/effects/fs-transaction.ts`

## P1 — Boundary and authority

`standard-plan.ts` is the single mutation plan consumed by both `adopt` and
`runInit()`. It reads the canonical assets, workflow contract, and target
state, then produces an ordered operation list. `FsTransaction` is the only
executor. Dry-run and apply therefore share the same operation model; there is
no shell apply path, second parser, or compatibility read in the runtime.

The plan covers scaffold/state files, typed hook projections, templates,
reference stubs, workflow archives, known-generated cleanup, deployment
scaffold, and managed-block updates. It refuses malformed managed JSON,
symlinks, unsafe paths, occupied move targets, and ambiguous ownership before
writing.

## P2 — One transaction trace

```text
adopt / runInit
  -> validate target and mode
  -> planAdoption (pure)
  -> preflight every operation and manifest path
  -> one FsTransaction apply
  -> persist manifest and registry projections
  -> refresh handoff / strict workflow evidence
```

The transaction creates a manifest under
`.ai/harness/backups/fs-transaction/<transaction>/manifest.json`. Writes save
their prior bytes and content hash; removes move the old file into the
transaction backup area; moves record source and destination. A later manifest
failure is a failed apply, never a success result. Rollback restores only
transaction-owned operations and refuses to overwrite a post-apply user edit.

## HRD-09 legacy retirement

The typed host-event cutover is represented in the same plan:

1. Retired generated hook assets are listed by exact relative path and
   `sha256:<64-hex>` fingerprint in the workflow-contract migration manifest.
2. `knownGeneratedFile()` removes a path only when its current bytes match the
   declared fingerprint. A mismatch emits a warning and preserves the file;
   wildcards and undeclared fingerprints are rejected.
3. Repo-local `.codex/hooks.json` and `.claude/settings.json` are parsed only
   by the explicit migration step. The managed command detector removes
   repo-harness-owned commands while preserving custom commands, unrelated
   events, and sibling hook blocks.
4. The removal, any `gitUntrack`, adapter rewrite, and typed helper projection
   are committed by the same `FsTransaction`. There is no post-apply shell
   cleanup and no runtime dual-read compatibility path.

The self-host source checkout is excluded from downstream generated-runtime
cleanup so its product source under `scripts/` is not mistaken for a copied
helper. Downstream adoption still retires only known generated bytes.

## Invariants

- **One source of truth**: assets and the workflow contract define canonical
  bytes and migration inventory; the plan, manifest, registry, and Markdown
  views are projections.
- **Fail closed**: malformed input, unsafe paths, hash mismatch, and ownership
  ambiguity stop or warn without broadening deletion scope.
- **Preserve user ownership**: custom hook siblings and non-matching files are
  not rewritten or deleted.
- **Atomic evidence**: the manifest is part of the same transaction boundary as
  the filesystem effects it describes.
- **No compatibility authority**: old host-event scripts are migration input
  only; host execution reaches the typed route registry.

## Recovery

`repo-harness adopt rollback --transaction <manifest>` uses the recorded
operation list. It restores backups and Git index state only when the target is
still transaction-owned. A changed target, occupied restore path, symlink, or
invalid manifest produces a structured rollback failure rather than overwriting
user data.
