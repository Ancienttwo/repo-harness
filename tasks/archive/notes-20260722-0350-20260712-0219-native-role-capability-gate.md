## Design Decisions

- Treat OpenAI's released `SubagentStart` fields (`turn_id`, `agent_id`, `agent_type`, and `model`) as the only runtime authority. Do not infer support from CLI version, task names, installed TOMLs, or rollout transcripts.
- Store one atomic JSON observation per child, keyed by a SHA-256 of `turn_id + agent_id`. Shared scalar state was rejected after architecture/security review demonstrated that a later verified sibling could overwrite an earlier default-role or mismatch observation.
- Aggregate every sibling deterministically with `invalid > mismatch > unavailable > unverified > verified`. A newly initialized delegation with no observations retains the latest completed canary so an empty prompt reset cannot erase authoritative negative evidence.
- Enumerate project custom-agent TOMLs before user TOMLs and select by schema-authoritative `name`; filename equality is only a convention. Duplicate names, malformed/ambiguous configuration, unsafe symlinks, and malformed authoritative hook fields fail closed as `invalid`.
- Allow `model` to be absent on unrelated valid profiles, matching the official optional field. The selected profile must pin a bounded model before model routing can be verified.
- Bind verified/mismatch observations to the selected TOML SHA-256 and re-check the current file during readiness reporting. A changed, deleted, symlinked, or relocated-outside-authority config invalidates the old canary.
- Keep missing evidence advisory `unverified`, as required by the contract. This supports first-run bootstrap; it does not prove native role routing.
- 2026-07-13: fixed a TOCTOU race in `subagent-start-context.sh`'s final commit. `resolveStatePath` only validates that `latest.json`'s `scope_id` matches this invocation's own scope at read time; the per-turn `state` then comes from a separate scoped file (`turns/<scope>.json`) that a concurrent invocation never touches, so it stays correct. But the old code then wrote that in-memory `state` to `latest.json` unconditionally, so a brand-new delegation (a fresh advisor `UserPromptSubmit` call for a different turn) that claimed `latest.json` in the interim got silently clobbered back to the stale turn. Fix: immediately before the `latestPath` write, re-read the on-disk `latest.json` and only overwrite it when its `scope_id` still equals `state.scope_id`; otherwise skip that write and leave the newer delegation's pointer alone. The `statePath` write stays unconditional since it is always scoped to this invocation alone and cannot be clobbered by a sibling turn.
- 2026-07-13 (second round): an independent Codex review of the fix above found it still incomplete: the re-read-and-compare and the following `atomicWriteJson` (create temp file, write, rename) are two separate steps, so a fresher write from another invocation could still land in the gap between the compare and the rename and be clobbered by this invocation's rename. Closed for real with a dependency-free mutual-exclusion lock rather than narrowing the window further: `fs.mkdirSync` on a sibling `latest.json.lock` directory is a single atomic syscall that fails with `EEXIST` when already held, so it doubles as a lock with no npm lockfile package and no native flock binding (`assets/hooks/subagent-start-context.sh`, `acquireLock`/`releaseLock`/`sleepMs`, wrapping the read-compare-write sequence around `paths.latestPath`). Stale-lock reclaim uses an 8-second mtime threshold (SubagentStart hooks are short-lived, so a lock older than that means a crashed holder); the acquire loop is bounded to ~2 seconds total (25ms backoff) and, if it cannot acquire, skips the `latestPath` write entirely rather than hang the hook — a skipped write is safe, a hung hook is not. `sleepMs` uses `Atomics.wait` on a throwaway `SharedArrayBuffer` for a real synchronous sleep (this script has no event loop to `await` inside), falling back to a busy-wait loop if unavailable.
- 2026-07-13 (second round, scope note): before trusting that the lock alone closed the race, ran a disposable probe reproducing Codex's exact scenario (a real concurrent `codex-delegation-advisor.sh` call as "turn B", paused via the same rendezvous-splice technique as the regression test) against the lock-only fix. It did not close the race: `codex-delegation-advisor.sh:182` wrote `latest.json` with a plain unconditional `fs.writeFileSync`, no lock, no compare, no atomic rename. A lock confined to `subagent-start-context.sh` cannot stop a writer that never checks it — mutual exclusion requires every writer of the shared resource to hold the same lock, or it provides no protection at all against the ones that do not. The probe confirmed this empirically: with the lock only in `subagent-start-context.sh`, the advisor's fresh write for turn B landed while turn A held the lock, and turn A's queued write then clobbered it back to turn A regardless — the exact bug, unfixed. This is a scope call worth flagging explicitly: the task that produced this fix named only `subagent-start-context.sh`, but the fix cannot satisfy its own stated goal ("close this properly ... Codex will find it again") without also gating `codex-delegation-advisor.sh`'s write to the same `latest.json.lock`, so that file was extended with the identical `acquireLock`/`releaseLock`/`sleepMs` helpers and its single `fs.writeFileSync(latest.json, ...)` call wrapped the same way. Both files were already inside this contract's `allowed_paths` from the first round, so this stays within the approved contract boundary even though it goes one file beyond the literal second-round dispatch text; re-ran the same probe after the extension and confirmed the advisor now blocks while turn A holds the lock and correctly writes turn B's pointer once turn A releases it (chronologically correct: whichever critical section starts later wins, matching what a "latest" pointer should mean).
- 2026-07-13 (second round, test design): the regression test needed to interleave a competing write strictly between turn A's compare-passed moment and its rename, not before the compare (the first-round test already covers that earlier window). A fixed-delay race between an async advisor call and a paused turn A turned out to be genuinely flaky against the pre-fix code — the advisor's unlocked write is fast but not instant (real child-process startup), so whether it lands before or after turn A's post-release write is a real timing coin flip, not a fixed-order guarantee. Fixed by polling for observed evidence that the advisor's write already landed (bounded to 1.5s) before releasing turn A, instead of guessing a delay: this makes the pre-fix failure reproduce deterministically (turn A is only released after turn B's write is confirmed on disk, so turn A's later unconditional write, if the code is broken, is guaranteed to come after and clobber it), while remaining a genuine real-process race on the fixed side (the poll simply times out there, since the advisor is legitimately blocked by the lock turn A holds). A second test repeats the same real-concurrency race across 10 independent temp repos with a varying release delay, with no deterministic poll gate, specifically to prove the fix holds under real, non-deterministic process scheduling and not just one lucky interleaving.
- 2026-07-13 (third round): an independent Codex review of the second-round mkdir lock found a real remaining bug: stale-reclaim in `acquireLock` only checked `mtime > 8s`, with no check of whether the current holder was actually still alive, and `releaseLock` unconditionally `rmdirSync`d the lock directory with no ownership check. Concrete failure: holder A pauses (GC, system load, a slow subprocess) past the 8s stale threshold while still legitimately inside its critical section; waiter B sees the stale mtime, reclaims by deleting A's lock directory, and proceeds under its own (correct, from B's perspective) critical section; A eventually resumes, still believes it holds the lock, clobbers whatever B legitimately wrote, then calls the old unconditional `releaseLock`, which deletes B's lock directory (or a third process C's, if C has since acquired) regardless of current ownership. Fixed by adding an ownership token: `acquireLock` now writes an `owner.json` marker (`{ pid, token, acquired_at }`) inside the lock directory immediately after `mkdirSync` succeeds, and returns `{ acquired, token }` instead of a bare boolean (both call sites in `subagent-start-context.sh` and `codex-delegation-advisor.sh` updated to read `.acquired` and pass `.token` through to `releaseLock`). `releaseLock(lockDir, token)` now reads the current `owner.json` first and only removes the lock (via `fs.rmSync(lockDir, { recursive: true, force: true })`, since the directory is no longer empty once it holds a marker file — the old bare `rmdirSync` would now fail with `ENOTEMPTY`) when the token still matches; a mismatch or missing marker means the caller no longer owns the lock, so it is left untouched. Stale-reclaim in `acquireLock` now additionally calls a new `isLockOwnerAlive(lockDir)` helper before reclaiming: it reads `owner.json` and probes `process.kill(pid, 0)`, treating a clean call or `ESRCH` as "gone" (safe to reclaim) and `EPERM` as "still alive, just not signalable" (do not reclaim); a missing or corrupt marker is also treated as abandoned (the only way a stale lock can lack a marker is a crash in the short window between `mkdirSync` and the marker write, which is itself evidence of abandonment). Reclaim now requires BOTH the mtime-stale condition AND a confirmed-dead (or marker-less) owner, per the dispatch's explicit requirement — a dead-but-not-yet-stale lock is still left alone, matching the existing bounded-retry/skip-not-hang contract. Kept fully dependency-free (no new npm package); the ~2s bounded total acquire retry and skip-not-hang-on-timeout behavior are unchanged. Both hook files (plus their `assets/hooks` mirrors, which are the actual canonical projection source per `scripts/sync-hook-sources.ts` — `assets/hooks` projects onto `.ai/hooks`, not the reverse, confirmed by reading the sync script before editing) received the identical implementation; `bun run sync:hooks` / `bun run check:hooks` confirm byte-identical parity after the edit.
- 2026-07-13 (third round, test design): the two new regression tests exercise `acquireLock`/`releaseLock` directly rather than through the full hook CLI route, since the dispatch specifically asked to lower `staleMs` via the lock options for a fast, deterministic test instead of sleeping past the real 8s default, and the existing full-script race tests have no way to pass a custom `staleMs` through to the real invocation. A small test-only harness (`extractLockFunctionSource` / `buildLockHarnessModule` in `tests/cli/hook.test.ts`) extracts `sleepMs`/`acquireLock`/`isLockOwnerAlive`/`releaseLock` verbatim (by brace-counting from each `function <name>(` marker) out of the real `assets/hooks/subagent-start-context.sh` and wraps them in a tiny CLI (`hold` / `attempt` / `die-holding` modes) so each side of the race is a genuine separate OS process, not an in-process simulation. `isLockOwnerAlive` is extracted only when present and a `normalizeAcquire` shim accepts both the old boolean return and the new `{ acquired, token }` shape, so the same harness builder also drives a pre-fix source snapshot for the fail-then-pass demonstration without needing a second harness. The "live holder" test spawns holder A (real synchronous `sleepMs`-based hold, not a busy-wait, so it is a genuinely live-but-blocked process) for 1000ms, waits `staleMs(150)+100ms` so the lock's mtime has actually aged past the stale threshold before the contender looks at it, then gives contender B a bounded 300ms attempt — comfortably inside A's remaining hold window — and asserts B never acquires. The "dead holder" test spawns a real child in `die-holding` mode that acquires and exits immediately without releasing (simulating a crash mid-critical-section), waits for the real process `close` event so the pid is confirmed exited (not merely paused), double-checks liveness itself via `process.kill(pid, 0)` before proceeding, then asserts a fresh attempt after the stale threshold successfully reclaims.
- 2026-07-13 (third round, fail-then-pass): reproduced the exact bug with real captured output before trusting the fix. Temporarily overwrote `assets/hooks/subagent-start-context.sh` (the file both new tests read from) with a saved pre-third-round snapshot (keeping the fixed version backed up separately) and re-ran only the new "live holder" test: it failed with `Expected: false, Received: true` at the `expect(attemptResult.acquired).toBe(false)` assertion — i.e., contender B's harness process genuinely acquired the lock while holder A's harness process was still alive and sleeping, reproducing Codex's exact scenario end to end with two real OS processes. Restored the fixed file from the backup (confirmed byte-identical via `diff`), re-ran `bun run sync:hooks`/`bun run check:hooks` (same projection digest `e1c02f2c...` as before the revert, confirming an exact restore), then re-ran the "live holder" test 3x: pass all three times. The "dead holder" reclaim test was also run against the reverted pre-fix code as a sanity check (not a required fail case, since old code has no liveness concept at all and just reclaims unconditionally on stale mtime): it passed there too, confirming the new test does not regress a case the old code already handled, only adds the missing liveness guard.
- 2026-07-13 (fourth round, replacement not a patch): a fourth independent Codex review of the third-round mkdir-plus-owner.json lock found two more real races, and per explicit user decision this round stops patching that design (three prior rounds each found a new TOCTOU gap in the same non-atomic multi-step primitive) and replaces the mechanism outright: (1) `mkdirSync(lockDir)` succeeding and the separate `owner.json` write were not atomic together, so a holder could pause in exactly that gap long enough to look stale-and-markerless, get reclaimed by a waiter, and then have its own marker write land inside the reclaimer's new directory, silently overwriting the reclaimer's marker so both believed `acquired:true`; (2) the stale-reclaim path itself (check owner is dead, then delete the directory, then mkdir again) was its own check-then-act sequence, so two concurrent reclaimers could race such that one deletes the lock the other had just legitimately recreated. Replaced the directory-plus-separate-marker-file design with a single lock FILE per `${latestPath}.lock` path, acquired via `fs.writeFileSync(lockPath, JSON.stringify({pid, token, acquired_at}), { flag: "wx" })`: `wx` is `O_CREAT | O_EXCL | O_WRONLY`, one call that atomically creates and opens the path (throwing `EEXIST` synchronously if it already exists) and writes the small owner-marker payload as part of that same successful open, so there is no intermediate state where the lock exists but has no readable owner yet -- this closes race (1) structurally, not by narrowing the window further. This assumes the marker payload stays small and the hook always runs against a local or tmpfs-backed filesystem, so that lock existence and lock content are effectively one fact established by a single syscall; it does not attempt to solve the general case of multi-writer torn writes for a larger payload or a networked filesystem (documented explicitly in the `tryCreateLockFile` comment). Stale-reclaim keeps the same shape as before (mtime-stale AND confirmed-dead-or-unreadable owner via `isLockOwnerAlive`, itself adapted to read the lock file directly instead of a `owner.json` inside a directory), but on a reclaimable lock now unlinks and immediately retries the `wx` create in the SAME loop iteration (not deferred to the outer loop's backoff); if that retry also throws `EEXIST` (a different creates own wx call won the race in the gap between this unlink and this retry), the call does not assume ownership -- it falls through to the shared bounded sleep/retry and the next full iteration re-reads and re-evaluates staleness from scratch. The one and only correctness property for ownership is: a call believes `acquired:true` if and only if its OWN `tryCreateLockFile` call (the initial attempt or the reclaim retry) returned `created:true` -- never from having merely performed a preceding unlink. Kept dependency-free (no npm package, no native flock binding), the same 8s default staleMs and ~2s default bounded acquire retry, and both hook files (`subagent-start-context.sh`, `codex-delegation-advisor.sh`) plus their `assets/hooks` mirrors received the identical implementation, synced via `bun run sync:hooks` and confirmed with `bun run check:hooks`.
- 2026-07-13 (fourth round, first attempted fix was insufficient -- content verification does not create exclusivity): while building the required new regression test (two concurrent reclaimers racing a real, forced rendezvous against the same stale-and-dead lock -- see the test-design entry below), the first version of the wx-based redesign above -- with a content-blind `fs.unlinkSync(lockPath)` in the reclaim path, exactly as most naturally follows from "check owner is dead, then delete, then wx-create again" -- reliably reproduced race (2) empirically: `Received: 2` (both racers believed `acquired:true`) on the very first trial run, and across a larger sample failed roughly 2 of every 5 runs. Root cause: `unlinkSync` does not check what it is deleting; if racer A completes its own unlink-then-wx-create fully (creating a fresh, live, legitimate lock) before racer B's own unlink call fires, B's unlink destroys A's fresh lock instead of the stale thing B evaluated, and B's subsequent wx-create then succeeds on the now-empty path, so B also gets `acquired:true` while A never released and still believes it holds the lock. First attempted fix: capture a snapshot (`fs.readFileSync`) of the lock content at the moment the `reclaimable` decision is made, and immediately adjacent to the `unlinkSync` call re-read the path and require the content to still be byte-for-byte identical before unlinking. An initial 5/5-pass sample looked like it had closed this, but a larger 15-run sample immediately afterward failed 3/15 with the same `Received: 2` output -- the content-verification step does NOT create exclusivity, it only adds one more read; two racers running the identical code at the identical forced starting point routinely complete that re-read while the original content is still unchanged for BOTH of them, before either has acted, so both still proceed to the destructive step together. This was a real lesson in this round: narrowing a check-then-act window by adding another check-then-act step of the same kind does not close a concurrency bug, no matter how small the window looks on paper -- only a genuinely exclusive operation does. This confirms the exact discipline the dispatch was asking for (do not keep patching a non-atomic sequence; use a real atomic primitive) applies recursively to fixes attempted within this same round, not only to the retired mkdir mechanism.
- 2026-07-13 (fourth round, actual fix -- exclusive reclaim marker): closed for real by demoting "two reclaimers racing the same destructive unlink-then-recreate step" to an ordinary contended `wx` create, which the OS already guarantees has exactly one winner. Before touching the main lock destructively, a reclaimer must first win an exclusive marker at `${lockPath}.reclaiming`, created with the identical `tryCreateLockFile` primitive as the main lock itself. Only the process whose own marker-create call returns `created:true` proceeds to unlink and retry-create the main lock (in a `try`/`finally` that releases the marker afterward via the existing token-verified `releaseLock`, not a raw unlink, regardless of whether the main-lock reclaim succeeded). A process that loses the marker create does not touch the main lock on that pass at all -- it only reclaims the marker itself if the marker looks mtime-stale AND confirmed-dead (the same two-part test as the main lock), which is accepted as an extremely low-probability path since the marker is only ever held for a handful of synchronous, non-sleeping syscalls with no sleep in between, so a genuinely abandoned marker implies a hard kill inside that tiny window. The main lock's ownership property is unchanged and still holds through every return path: `acquired:true` only from this call's own `tryCreateLockFile` succeeding (the initial attempt, or the reclaim retry reached only after winning the marker), never from having merely performed a preceding unlink. Verified far more rigorously this time given the first fix's false confidence: 50 repeated runs of the race test (15, then 20, then a final confirming 15) all passed -- 0 failures across 250 forced-race trials (each run loops the test 5 times internally). This is a categorically different guarantee from the content-verification attempt: exclusivity is now provided by the same OS-level `O_EXCL` atomicity already trusted for the main lock, not by racing two independent reads against each other.
- 2026-07-13 (fourth round, test design): the new test (`two concurrent reclaimers racing the same stale-and-dead lock never both believe acquired:true`) forces the exact adversarial interleaving deterministically instead of hoping normal OS scheduling happens to overlap two spawned processes. It splices a rendezvous barrier into the extracted `acquireLock` source (via the same verbatim brace-counting extraction harness used by the two existing lock tests; no new function name needed extracting, since the marker-based fix reuses `tryCreateLockFile`, `isLockOwnerAlive`, and `releaseLock`, all already in the harness's extraction list) immediately after each racer independently observes `EEXIST` on its own first create attempt and before either touches the lock file: both racers write a pid-named marker into a shared "reached" directory, then busy-wait (`Bun.sleepSync`) on a shared go-file. The parent test polls until both markers exist, then writes the go-file, releasing both racers to run the reclaim sequence at effectively the same instant. Each racer is also given `totalTimeoutMs: 1`, far shorter than the rendezvous pause itself: `acquireLock` computes its deadline before the pause even begins, so by the time a racer resumes and finishes its one reclaim attempt, the deadline has already passed and the loop returns immediately instead of looping back for a second, ordinary retry. This isolates the exact property under test -- the outcome of ONE forced-concurrent reclaim race -- from ordinary sequential re-acquisition (a racer legitimately acquiring later, after the other has released), which is expected, healthy mutual-exclusion behavior, not the bug under test; without this isolation, a naive "did each racer ever succeed" assertion would be met by both racers in the common case (they just take turns), which is correct behavior, not evidence either way about the specific race. The test asserts exactly 1 (not merely <= 1) of the two racers reports `acquired:true`, since a genuinely mutually-exclusive reclaim of a real stale-and-dead marker guarantees exactly one winner. Repeated 5 times internally (fresh scratch dir and fresh lock file each trial) to rule out a single lucky interleaving, matching the round-2 many-real-concurrent-trials rigor; this internal 5-trial loop is what first exposed the content-verification attempt's 3/15 real-world failure rate once run enough times across enough separate invocations, not a single 5/5 sample, which is why this round's overall verification leaned on dozens of repeated invocations rather than trusting one clean batch.
- 2026-07-13 (fourth round, fail-then-pass): a direct fail-then-pass against the third-round mkdir-based mechanism was not attempted for this specific test, because that source has no `tryCreateLockFile` / `attempt.created` shape for the same splice anchor to target -- proving the point that way would need a second, differently-anchored splice tailored to the old mkdir-plus-rmSync structure, which the dispatch's stated alternative ("confirm it passes cleanly...across multiple repeated runs (5+)") does not require. Instead, fail-then-pass was demonstrated against the actual naive content-blind-unlink baseline (the first attempt, before either fix): temporarily reverted just the reclaim step in `assets/hooks/subagent-start-context.sh` to a plain unconditional `unlinkSync` (keeping the marker-based fix backed up separately) and ran the new test repeatedly. Honest empirical results, not oversold: the naive baseline failed 2/8 runs with real captured output (`Expected: 1, Received: 2`), consistent with the roughly 20-40% failure rate observed earlier against this same baseline. Restored the marker-based fix from the backup (confirmed byte-identical via `diff`, and via an identical `bun run check:hooks` projection digest before and after the revert/restore cycle), then re-ran the test 15 more times: 15/15 pass (75 more forced-race trials, all resolving to exactly one winner), on top of the 35 already-passing runs (175 trials) recorded in the entry above -- 50 total clean runs, 250 total forced-race trials, 0 failures, against the final marker-based fix.
- 2026-07-13 (fifth round, drop reclaim entirely -- explicit user decision, final round for this lock): a fifth independent Codex review of the fourth-round exclusive-reclaim-marker design found it STILL had two residual races nested one layer deeper: (a) `writeFileSync(lockPath, ..., { flag: "wx" })` is technically `open()` then `write()` as two syscalls, so an OS preemption between them could theoretically be misread as abandonment; (b) the reclaim-marker's own stale-marker cleanup path did a raw `unlinkSync` with no verification it was still deleting the same content it judged stale, so it could delete a different process's brand-new legitimate marker. Five consecutive independent review rounds each finding a new TOCTOU race one layer deeper than the fix that closed the previous one (no lock -> mkdir-one-site -> mkdir-both-sites-no-ownership -> ownership-token-with-non-atomic-mkdir-plus-marker -> single-file-wx-with-racing-reclaim -> reclaim-marker-with-unverified-stale-cleanup) is not evidence that this round's fix is finally the deepest layer; it is a decisive, structural signal that safely reclaiming a lock from a possibly-crashed holder using only bare filesystem primitives (no flock, no new dependency) cannot be made fully correct through incremental patching -- every fix closes the reported race and opens an equivalent one nested one layer deeper in the mechanism that protects the previous fix, and there is no reason to expect a sixth round would behave differently. Per explicit user decision, this is the final round for this lock: the stale-lock reclaim feature is dropped entirely rather than patched a fifth time, and no further reclaim/staleness/ownership-verification-on-reclaim logic may be re-added to this mechanism.
- 2026-07-13 (fifth round, the simplified mechanism): `tryCreateLockFile` is unchanged -- it was never the source of any of the five findings; exactly one `wx` (`O_CREAT|O_EXCL|O_WRONLY`) create can ever succeed for a given path, full stop. `acquireLock(lockPath, options)` is now: try `tryCreateLockFile`; if `created`, return `{ acquired: true, token }`; if it throws `EEXIST`, do NOT check staleness, do NOT check owner liveness, do NOT attempt any reclaim -- just check the bounded deadline (still ~2s default `totalTimeoutMs`, still ~25ms default `retryDelayMs`, both options unchanged in shape) and either return `{ acquired: false, token: null }` or sleep and retry. `isLockOwnerAlive` is deleted outright (it has no callers left). The reclaim-marker path (`${lockPath}.reclaiming`, the nested `tryCreateLockFile`/`releaseLock` calls around it, and its own stale-marker cleanup) is deleted outright. All `mtimeMs`/`staleMs` reads and comparisons are deleted outright -- `acquireLock` no longer calls `fs.statSync` at all. `releaseLock(lockPath, token)` is intentionally UNCHANGED in logic (read the file, compare the token to what this call's own `tryCreateLockFile` returned, unlink only on a match): the dispatch confirmed this was never the source of a live bug (the residual there -- PID reuse of an unrelated exited process, or a torn write -- is the same already-accepted, extremely-low-probability category documented since round 3/4, not a new finding), and removing the token check in favor of an unconditional unlink would introduce a new hazard (a caller releasing a lock it no longer owns after a legitimate handoff) that the five review rounds never actually found a problem with. Only the doc comments around `acquireLock`/`releaseLock` were rewritten (the code paths inside `releaseLock` are byte-for-byte identical to round 4) to state the design choice plainly: no reclaim is attempted; a lock left behind by a crashed holder stays held until removed by hand (for example, `rm .ai/harness/delegation/latest.json.lock`), accepted because `SubagentStart`/`UserPromptSubmit` hooks are short-lived and this lock only guards the shared `latest.json` convenience pointer, never the isolated per-turn state write (which always succeeds regardless of this lock's outcome, unchanged from every prior round). Two now-stale inline comments inside the retained `releaseLock` body (`"reclaimed as abandoned"`, `"reclaimed as stale"`) were also corrected to stop asserting a concept that no longer exists anywhere else in the file, without touching the surrounding logic. Applied identically to `assets/hooks/subagent-start-context.sh` and `assets/hooks/codex-delegation-advisor.sh`; synced to `.ai/hooks/` via `bun run sync:hooks` (new projection digest `sha256:842c6bd5f0c112920aa9b4885de4845261b118fc854b10d2804fd7dc7d785f3a`) and confirmed with `bun run check:hooks`. While drafting the new comments, the same apostrophe-truncation gotcha recorded in the 2026-07-13 entry near the end of this file recurred twice (`process's`, `lock's`) and was caught immediately by `bash -n` failing with a syntax error at the exact line, then fixed by rephrasing both sentences to avoid the apostrophe rather than escaping it (this `bun -e '...'` body runs inside a single-quoted bash argument, so a literal apostrophe silently truncates the argument well before the real closing quote).
- 2026-07-13 (fifth round, tests): removed the two round-4/round-5 tests that specifically exercised behavior this round deletes -- `acquireLock reclaims a stale lock once its recorded owner process has actually exited` (reclaim-from-a-dead-holder no longer exists) and `two concurrent reclaimers racing the same stale-and-dead lock never both believe acquired:true` (there is no more reclaim step for two reclaimers to race through). Adapted `acquireLock does not let a concurrent attempt steal the lock from a still-live holder past the stale threshold` -> renamed to `acquireLock never lets a concurrent attempt acquire while the lock file still exists`, dropping the `staleMs`/liveness-check framing (the simplified lock has no staleness concept left to reference) while keeping the exact same two-real-process mechanic: holder A holds via a real synchronous sleep, contender B's bounded attempt during that hold must return `acquired:false`, A's own release afterward must still work. ## Tradeoffs Considered

| Option | Decision | Reason |
|--------|----------|--------|
| Parse Codex rollout/session JSONL | Reject | OpenAI documents transcript internals as unstable; it would also couple repo readiness to unrelated personal session state. |
| Treat installed agent files or responsibility labels as routing proof | Reject | The 0.144.1 canary produced `agent_role:null`/`default` and inherited the root model despite installed custom profiles. |
| Shared scalar or shared map in delegation state | Reject | Last-writer-wins and unlocked read-modify-write lose sibling failures. Independent atomic files avoid cross-child contention. |
| Skip a malformed unrelated TOML | Reject / fail closed | An unparseable file cannot prove that its hidden `name` is unrelated to the selected `agent_type`; accepting the directory would manufacture certainty. |
| Make no-canary evidence fail strict readiness immediately | Reject | The approved contract explicitly keeps missing evidence advisory; authoritative negative, invalid, malformed, or drifted evidence does fail strict. |
| Patch the fourth-round reclaim-marker design a fifth time | Reject (2026-07-13, explicit user decision) | Five independent review rounds each found a new TOCTOU race one layer deeper than the previous fix, always nested inside the mechanism that protected the prior round's fix. That pattern is a decisive signal that incremental patching of a bare-filesystem-primitive reclaim mechanism does not converge, not a coincidence to keep chasing. |
| Drop stale-lock reclaim entirely vs. add a real fd-based lock (flock) | Reject the flock route; drop reclaim instead | Adding a new dependency or a native binding was explicitly out of scope across every round of this lock (dependency-free was a standing constraint). Between "add a dependency" and "stop trying to reclaim a shared convenience-pointer lock at all," the latter is the smaller, more honest change given the guarded resource is disposable (a crashed holder's lock only blocks a `latest.json` pointer write; the isolated per-turn state write is unaffected either way). |
