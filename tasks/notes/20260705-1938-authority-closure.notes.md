- **T1 target content**: unified all 5 contract-template copies to exactly match canonical `assets/templates/contract.template.md`. - **Status: Pending vs Active resolved as drift, not a deliberate split**: per plan instruction to read the pipeline logic before deciding. Grepped `scripts/plan-to-todo.sh`, `scripts/ensure-task-workflow.sh`, `scripts/lib/project-init-lib.sh` for any `Pending`→`Active` substitution; found none — `render_contract_file`'s `sed` pipeline only substitutes `{{PLACEHOLDER}}` tokens, never touches literal `Pending`/`Active` text. The other "Active" grep hits in these files belong to a *different* template entirely (`implementation-notes.template.md`, which always seeds `Active`); the "Pending" hits besides the contract seed belong to `review.template.md`'s seed. Neither is evidence of a deliberate contract-status pipeline. Concluded the seeds' `Pending` was stale drift from before the canonical template was updated to `Active`, and unified all three seeds to `Active`.
- Renamed the dead `bun run typecheck` command to `bun run check:type` (real script name, `package.json:38`) across all 5 copies, per the plan's already-decided rename.
- Full diff inventory (A vs B vs C vs D vs E, pairwise) confirmed the plan's own reconnaissance was accurate: no undocumented/unexpected logical differences surfaced beyond what section T1 already named. No STOP triggered.
- **T2 Falsifier section** inserted verbatim (exact plan wording) immediately after `## Stop Conditions` in all 5 copies: `assets/templates/contract.template.md`, `.claude/templates/contract.template.md`, and the three embedded seed heredocs. All copies re-verified byte-identical/body-identical to canonical after the insertion (same method as T1).
- **Critical bash 3.2 parser bug found and fixed in `scripts/lib/project-init-lib.sh`**: inserting the Falsifier sentence (which contains exactly one apostrophe, in "task's") into `PI_TEMPLATE_CONTRACT`'s heredoc body broke both `bash -n` *and actual execution* (exit code 2, `unexpected EOF while looking for matching` a single quote) — not a lint false positive. Root cause, isolated with a 5-line minimal repro: on this system's `/bin/bash` (GNU bash 3.2.57, the stock macOS system bash — the only bash on this machine), a heredoc with a **single-quoted delimiter** (`<<'EOF'`, which should make the body 100% literal) nested inside a `$(...)` command substitution still gets its body scanned by `$(...)`'s boundary-matching lexer for quote characters. An **odd number of apostrophes** in the body confuses that scan into thinking a `'`-string was opened and never closed, so it reads past the real closing `)` looking for a matching quote until true EOF. Verified empirically: 1 apostrophe inside `VAR=$(cat <<'EOF' ... EOF\n)` fails; 2 apostrophes (even) pass; the same heredoc *without* the `$(...)` wrapper (plain `cat <<'EOF' ... EOF`, no capture) passes regardless of apostrophe count. Audited every other `$()`-wrapped heredoc in this file (`PI_DEFAULT_GITIGNORE_CONTENT`, `PI_DEFAULT_RUNTIME_ENTRIES`, `PI_EXTERNAL_TOOLING_HOSTS_DEFAULT`, `PI_TEMPLATE_RESEARCH`, `PI_TEMPLATE_SPEC`, `PI_TEMPLATE_PLAN`, `PI_TEMPLATE_REVIEW`, `PI_TEMPLATE_IMPLEMENTATION_NOTES`) — all have zero apostrophes, so this was a new risk introduced by this edit, not a pre-existing latent bug. This matters beyond a lint nuisance: `project-init-lib.sh` is `.`-sourced by `create-project-dirs.sh`, `migrate-project-template.sh`, and `init-project.sh`, all of which run under `set -e`/`set -euo pipefail` — a plain syntax error here breaks all three at the point they source this file, silently, on any Mac using the stock system bash.
  - Considered `read -r -d '' PI_TEMPLATE_CONTRACT <<'EOF' ... EOF` as an alternative fix (also passes `bash -n` and preserves content) but rejected it: `read -d ''` always returns exit 1 at heredoc EOF (there is no NUL byte to satisfy `-d ''`), and all three calling scripts run under `set -e`, so that single `read` would abort the sourcing script every time. Confirmed the temp-file fix has no equivalent nonzero-exit step.
  - Fix applied (scoped to only this one variable, no other `PI_TEMPLATE_*` needed it): `PI_TEMPLATE_CONTRACT_TMP="$(mktemp)"` then `cat > "$PI_TEMPLATE_CONTRACT_TMP" <<'EOF_TEMPLATE_CONTRACT'` (unchanged body) `EOF_TEMPLATE_CONTRACT`, then `PI_TEMPLATE_CONTRACT="$(cat "$PI_TEMPLATE_CONTRACT_TMP")"; rm -f "$PI_TEMPLATE_CONTRACT_TMP"`. This is the same `mktemp` idiom already used elsewhere in this file (`tmp_file="$(mktemp)"` at the pre-existing lines ~818, ~1760-1761), so it matches existing style. Verified: `bash -n` clean, actual sourcing produces the byte-identical variable value (diffed against `ensure-task-workflow.sh`'s equivalent, still byte-identical), Falsifier sentence intact with its apostrophe, and the temp file is not leaked.

## Deviations From Plan Or Spec

- Two `tests/helper-scripts.test.ts` fixtures (`contract-worktree finish should require external acceptance...` at the fixture package.json around line ~1504, and `ship-worktrees default mode should finish without merging...` around line ~1661) hardcoded a synthetic `package.json` with a `scripts.typecheck` key to satisfy the old exit-criteria command. Renaming the template's exit-criteria command to `bun run check:type` (per T1's already-decided rename) made these fixtures' rendered contracts require a `check:type` npm script that didn't exist in the fixture, so `contract-worktree.sh finish` / `ship-worktrees.sh` failed non-zero downstream. Verified via `git stash` that both tests passed on the pre-T1 baseline and failed only after the template edit, confirming this was a direct, expected consequence of the deliberate rename rather than an unrelated regression. Fixed by renaming both fixtures' `scripts.typecheck` key to `scripts["check:type"]` (same command value). Not explicitly named in the plan's T1 file list, but required to keep existing `tests_pass` coverage green under the plan's own rename decision; within `tests/`, an already-allowed path.
- **`scripts/lib/project-init-lib.sh` assignment mechanism for `PI_TEMPLATE_CONTRACT` restructured** (temp-file indirection instead of direct `$(cat <<'EOF'...)`) as a required, non-optional consequence of adding the plan's verbatim Falsifier sentence — see the bash parser bug above. Not named in the plan's T2 file list (which only anticipated inserting the section text), but the plan's own global rule 3 ("fail-closed, no compatibility fallback") and rule 7 ("run each slice's Verify and paste raw output") left no option but to fix a genuine `bash -n`-and-execution-breaking regression discovered by that same Verify step, scoped to the minimum needed (one variable, no unrelated heredocs touched).

## Tradeoffs Considered

| Option | Decision | Reason |
|--------|----------|--------|
| Make seed heredocs (C/D/E) match canonical A's full body vs. only patch the specific gaps recon named | Replace the full heredoc body with canonical content | Guarantees byte-level parity (not just heading-set parity) and is easier to keep correct than hand-patching ~6 separate gaps per file; still verified against the plan's named gap list before applying |
| Leave contract seed Status as `Pending` (seed-appropriate "not yet real" default) vs unify to `Active` | Unify to `Active` | No pipeline logic ever flips it later; a freshly-bootstrapped downstream repo would otherwise ship contracts permanently mislabeled `Pending`. Canonical template already treats `Active` as literal baked-in text, not a templated field |
| `project-init-lib.sh` apostrophe-in-`$()`-heredoc fix: `read -r -d ''` vs temp-file indirection | Temp-file indirection | `read -d ''` returns exit 1 at heredoc EOF by design; all 3 sourcing scripts run under `set -e`, so it would abort them. Temp-file approach's final step is still `$(cat "$tmpfile")`, which has the same trailing-newline-stripping semantics as the original `$(cat <<'EOF'...)`, so behavior is preserved exactly, not just content |
