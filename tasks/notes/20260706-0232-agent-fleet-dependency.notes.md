## Design Decisions

- P1: `fable_agents` inserted into `external_tooling` immediately after `hai_stack`, before `codex_automation_profile`, in all three faces (`.ai/harness/policy.json`, `scripts/ensure-task-workflow.sh` seed +mirror, `scripts/lib/project-init-lib.sh` `pi_write_harness_policy()` seed). Formatting follows each face's existing local convention rather than a uniform style: `.ai/harness/policy.json` uses the expanded (one-array-item-per-line) style already used by `waza`/`hai_stack` there; the two seed heredocs use the compact single-line array style already used by their own `waza`/`hai_stack`/`codex_automation_profile` entries. This matches the pre-existing precedent (the same divergence already exists between policy.json and the seeds for every other key) rather than introducing a new formatting rule.
- P1: confirmed via direct read that `scripts/ensure-task-workflow.sh`'s `external_tooling` block (its own `POLICY_EOF` heredoc, only written when `.ai/harness/policy.json` is absent) and `scripts/lib/project-init-lib.sh`'s `EOF_POLICY` heredoc are textually identical for every existing sub-key (`waza`, `hai_stack`, `codex_automation_profile`, `diagram_design`, `gbrain`, `codegraph`) except `codegraph.install_mode` (which is already environment-specific: `target-aware-mcp` in both seeds vs `self-host-dev-dependency-with-global-mcp-opt-in` in the self-hosted policy.json). `fable_agents` was added with the exact same value in both seed heredocs (`"advisory"`) to preserve that established parity, diverging only in policy.json (`"auto-install-on-init"`), matching the codegraph precedent the plan named.
- P1: no test does a static byte-comparison between the `POLICY_EOF` (ensure-task-workflow.sh) and `EOF_POLICY` (project-init-lib.sh) heredocs directly — that parity is a manual authoring obligation, not machine-enforced. (The `helper-scripts.test.ts:467-468` guard the plan's recon cited is for a *different* heredoc pair — the `CONTRACT_TEMPLATE_EOF`/`EOF_TEMPLATE_CONTRACT` task-contract markdown template — not the policy JSON seed. Verified by direct read before relying on it; the plan's line-number citation for that specific guard does not apply to the policy heredoc.) Actual runtime coverage for the policy heredocs comes from: `tests/create-project-dirs.runtime.test.ts` (exercises `project-init-lib.sh`'s fresh-init path), `tests/migration-script.test.ts` (exercises both the fresh-migrate and preserve-overrides-merge paths, both ultimately sourced from the same `pi_write_harness_policy()` heredoc), and `tests/helper-scripts.test.ts` (multiple invocations of `ensure-task-workflow.sh` against repos with no pre-existing `policy.json`, which trigger its `POLICY_EOF` heredoc and then `JSON.parse` the result — this would fail on any JSON syntax error even though it doesn't assert `fable_agents` values specifically).
- P1: added explicit `fable_agents` field assertions to `tests/create-project-dirs.runtime.test.ts` (fresh-init, expects `install_mode: "advisory"`) and to both relevant blocks of `tests/migration-script.test.ts` (fresh-migrate-no-existing-policy, and preserve-explicit-overrides-while-merging-defaults — both also expect `"advisory"` since neither fixture pre-supplies a competing `fable_agents` override), following the existing per-key assertion convention used for `waza`/`hai_stack`/`codegraph`. This was treated as in-scope because the contract's Scope section explicitly lists these files under "测试扩展" (test extension), not merely "must keep passing".
- P2: implemented `scripts/install-agent-fleet.sh` in the same "bash shell -> `exec node/bun - <<'NODE_EOF'`" architecture as `scripts/check-agent-tooling.sh`, rather than a plain-bash script. Chosen because the script's core job (YAML-frontmatter parsing, TOML string generation with byte-exact escaping, file byte comparison) is fragile in pure bash and this repo already has precedent for reaching for an embedded-JS runtime when a helper needs real string/JSON handling; `curl` is still invoked via `spawnSync`, matching `fetchWazaUpstreamSkills()`'s pattern (`curl -fsSL --max-time <n> <url>`).
- P2: managed-agent list, `raw_base`, and both target directories are hardcoded constants inside the installer (matching the values declared in `policy.json`'s `fable_agents` entry) rather than read from the invoking repo's `.ai/harness/policy.json` at runtime. This mirrors the existing precedent where `check-agent-tooling.sh`'s `WAZA_*`/`HOSTS` constants are also hardcoded and independently kept in sync with policy.json by convention, not read from it — appropriate here since fleet installation is a global (`~/.claude`, `~/.codex`) operation independent of whichever repo happens to invoke the command.
- P2: frontmatter validation is stricter than a generic lookup table: `effort` must be exactly `"max"` and `model` must be exactly `"opus"` or `"sonnet"` (matching the plan's literal wording, not merely "any (model,effort) pair present in a map"). Any other value is fail-closed `invalid`, gating **both** the Claude and Codex output for that agent (not just the Codex/TOML side) since an unparseable frontmatter is treated as an untrustworthy whole file, not just a Codex-generation problem.
- P2: verified the mapping and the embedded EXECUTION_BOUNDARY text empirically by running the installer against the repo's own real fixtures (`.claude/agents/*.md` as `REPO_HARNESS_FLEET_SOURCE_DIR`, a scratch `HOME`) before writing the formal test suite: fresh install reproduced all three `.codex/agents/*.toml` golden files byte-for-byte on the first attempt (`diff -q` clean), confirming the (opus,max)->(gpt-5.5,xhigh), (sonnet,max)->(gpt-5.5,medium), `tools:` present->`sandbox_mode = "read-only"`, and `body + "\n\n" + EXECUTION_BOUNDARY` construction rules against real golden data, not just against my own re-derivation of the plan's mapping table.
- P2: exit-code contract implemented as: non-zero only when **no** agent was successfully processed (fetch + frontmatter validation both succeeded for at least one agent) **and** no target file already existed for any agent beforehand. A single fetch failure, a single invalid-frontmatter agent, or drift on existing files never trips a non-zero exit on their own — matching "partial success = partial install + summary report, init's auto-install must not hard-fail."
- P2: registered `install-agent-fleet.sh` in three places kept in sync by hand (`assets/workflow-contract.v1.json` `helpers.scripts`, mirrored byte-for-byte into `.ai/harness/workflow-contract.json`; and `pi_install_helpers()`'s default helper-name list in `project-init-lib.sh:1108`). Verified end-to-end that no runner code needed to change: `repo-harness run install-agent-fleet` (through `src/cli/index.ts` -> `src/cli/runtime/helper-runner.ts`'s `readContractHelpers()`) dispatched correctly against a fixture source dir and fake HOME on the first try.
- P2: extended the existing "execution boundary canonical sentence stays identical across its constant sources" test in `tests/workflow-contract.test.ts` by adding `scripts/install-agent-fleet.sh` to its `sources` array, rather than duplicating a new parity test — this is the repo's established pattern for this exact cross-surface-parity concern (three other surfaces were already listed there: contract worker prompts, the MCP codex-goal path, and the Codex delegation advisor hook). Also added a `helpers.scripts` `toContain("install-agent-fleet.sh")` assertion alongside the existing per-helper entries in the same file.
- P2: `tests/install-agent-fleet.test.ts`'s own boundary-parity test could not check the installer's *source text* for the fully-joined boundary string (the source only holds it as an array of separate paragraph literals, joined with `"\n"` at runtime) — fixed by asserting each paragraph individually appears in the installer source, and asserting the fully-joined canonical text appears in the *generated TOML output* (the stronger, functional half of the check). Caught and corrected via an actual `bun test` failure before considering the suite complete, not by inspection alone.

## Tradeoffs Considered

| Option | Decision | Reason |
|--------|----------|--------|
| Uniform JSON array formatting across all 3 policy faces vs match each face's existing local style | Match each face's existing local style | Every prior external_tooling key (waza/hai_stack/codegraph) already diverges in formatting between policy.json (expanded) and the seed heredocs (compact); introducing a new uniform style for only the new key would make `fable_agents` inconsistent with its siblings within the same file |
| Plain-bash vs bash-shell-wrapping-embedded-JS for `install-agent-fleet.sh` | Embedded-JS (check-agent-tooling.sh style) | TOML generation needs precise escaping/quoting and byte-exact multi-line string construction; pure bash heredoc string-building for this is exactly the fragile pattern `tasks/lessons.md`'s bash-3.2 quote-parity entry warns about, while Node's template literals and `fs`/`JSON` APIs make the byte-exact golden reproduction straightforward and it was in fact byte-identical on the first manual smoke test |
| Read `fable_agents.*` from the invoking repo's policy.json at runtime vs hardcode constants in the installer | Hardcode constants | Matches the existing check-agent-tooling.sh precedent (WAZA_* consts are hardcoded, not read from policy.json) and keeps a global (~/.claude, ~/.codex) installer decoupled from whichever repo happens to invoke it |
