## Design Decisions

- Regression 3 (`required` + current-valid-evidence via stub-validator seam) already passed pre-patch — old and new logic agree on that one input shape. The failing pre-fix evidence for Slice 1 covers regressions 1-2 (`.ai/harness/runs/slice1-prefix-failures.txt`, `PRE_FIX_EXIT=1`); regression 3's value is drift detection plus pinning the declaration-driven path.
- Behavior correction, not weakened assertion: in `tests/workflow-state-lib.test.ts`'s "helper unavailable" test, `checks.contract.status`/`failure_class` now expect `fail`/`contract_failure` because the evidence-requirements gate depends on the same missing lib — fail-closed is the intended semantics.
- Post-freeze allowlist (`compute_post_freeze_allowlist` in `scripts/contract-worktree.sh`) is the deduped, insertion-ordered union of: `plans/<plan_base>`, `plans/archive/<plan_base>`, the contract file, the review file, `tasks/current.md`, `tasks/todos.md`, `.ai/harness/active-plan` and `.ai/harness/active-worktree` (unconditional local-marker deletes; no-op unless a repo/fixture actually tracks them), `tasks/archive/contract-<timestamp>-<raw_slug>.md` and `tasks/archive/review-<timestamp>-<raw_slug>.md` (unconditional), plus `tasks/notes/<artifact_stem_or_raw_slug>.notes.md` and `tasks/archive/notes-<timestamp>-<raw_slug>.md` only when a notes file exists, plus `tasks/archive/todo-<timestamp>-<raw_slug>.md` only when `tasks/todos.md` has non-whitespace content, plus the sprint backlog path parsed from the plan's `Source Ref` only when it matches `sprint:*#*`. `timestamp` is a single `date +%Y%m%d-%H%M` call taken before archiving and reused for every archive-family prediction so it agrees with `archive-workflow.sh`'s own (later, separate) timestamp call.
- Allowlist plumbing: `run_merge_gate` (`scripts/contract-worktree.sh`) takes the allowlist as trailing positional args after `base_ref`/`goal_file` (`shift 2`), turns each path into a repeatable `--allow-post-freeze <path>` flag, and forwards them to `merge-gate.ts run`. `merge-gate.ts` collects the flags into `post_freeze_allowlist: string[]` (deduped + sorted) on the written receipt. `goal_sha256` is `sha256(goalContent)` of the goal file read once at `run` time and stored on the receipt; `readReceipt` requires it as a non-empty string (fails closed on older receipts missing it). Fix-round-1 attempted to activate `goal_sha256` as a verify-time predicate (re-hashing whichever post-freeze archive destination shares the frozen goal's own basename) but reverted it after the finish e2e fixture proved `archive-workflow.sh` rewrites the plan's Status line before archiving it, so the archived goal is never byte-identical to the frozen `goal_sha256` on any real run — see "Fix Round 1" (fix 3) below. `goal_sha256` therefore remains captured integrity evidence on the receipt, not yet an active verify-time predicate; the `tasks/todos.md` deferral for this is restored.

## Fix Round 1 — External Review Remediation (2026-07-16)

Five externally-reviewed findings against the Part B (freeze-before-archive) design, fixed with the design frozen by the review; implementation only, no redesign.

Previously this path (`ship-worktrees.sh`'s own `finish --no-merge --gate-base <ref>` invocation) committed L and returned success without ever re-verifying the receipt against it. The plain no-gate `--no-merge` path (Step 6) and the merge-back path (Step 7) are unchanged.
- **Fix 2 (P1, allowlist shape guard + reviewed allowlist)**: `scripts/merge-gate.ts:150-183` adds `classifyPostFreezePath` (a closed 14-shape classifier: `plans/<basename>.md`, `plans/archive/<basename>.md`, `plans/sprints/*.md`, `tasks/contracts/*.contract.md`, `tasks/reviews/*.review.md`, `tasks/notes/*.notes.md`, `tasks/current.md`, `tasks/todos.md`, the four `tasks/archive/{contract,review,notes,todo}-*.md` families, `.ai/harness/active-plan`, `.ai/harness/active-worktree`) and `validatePostFreezeAllowlistShape`, called in `run` on every `--allow-post-freeze` entry; any path outside the closed set fails closed with `exit 2` naming the path, so production/test/asset paths are structurally impossible to allowlist. `post_freeze_allowlist` is now included in the `request` object sent to the gate reviewer (`scripts/merge-gate.ts:786`, before `invokeClaude`) so the PASS/FAIL decision covers it; the receipt still records the same deduped+sorted list.

  - **Finding (discovered via the mandated finish e2e fixture, per plan P3 point 6 and the fix-round-1 brief's own instruction to verify byte-preservation against that fixture before shipping)**: the plan's stated assumption "archive is a byte-preserving move" is false for all four SEMANTIC families against the real `scripts/archive-workflow.sh` (out of this contract's `allowed_paths`, same boundary as fix 5) — `set_plan_status` (`scripts/archive-workflow.sh:167-183`) rewrites the plan's `> **Status**:` line in place before its `mv` to `plans/archive/`; `completed_archive_gate` (`scripts/archive-workflow.sh:142-158`) does the same for the contract's Status ("Active"->"Fulfilled"); and the contract/review/notes archival step (`scripts/archive-workflow.sh:426-451`) prepends a 5-line `> **Archived**/Related Plan/Outcome/Lifecycle/Parent Run ID` audit header before writing each to its archive destination. Implementing the destination-content-hash comparison and the `goal_sha256` active-predicate check exactly as specified therefore made every real `contract-worktree.sh finish` / `ship-worktrees.sh` invocation fail verify (proven directly: the "ship-worktrees default mode..." e2e test in `tests/helper-scripts.test.ts` failed with `post-freeze semantic destination content does not match the frozen source: plans/archive/plan-20260304-1450-demo.md` on the very first full-suite run after implementing fix 3 as specified).
  - **Resolution**: per the brief's own instruction ("make that family's archive byte-preserving rather than loosening verify — fail closed, never guess"), and since making `archive-workflow.sh` byte-preserving requires editing a file outside this contract's `allowed_paths` (removing the Status rewrite and the audit header would also delete a wanted feature, not fix a bug) and predicting its output shape inside `merge-gate.ts` would be a shadow parser of `archive-workflow.sh`'s own logic (forbidden by this repo's one-parser-per-datum principle) — the destination-content-hash comparison (`role === "dest"` branch) and the `goal_sha256` active-predicate check were held back rather than shipped broken. `post_freeze_semantic` is still recorded on every receipt (harmless, and ready for a future change scoped to include `scripts/archive-workflow.sh`); the SEMANTIC live-path-removal check (`semanticLiveIsRemoved`) stayed active because it is true regardless of the destination's exact bytes — the live path genuinely disappears (as a delete, or the source side of a same-family rename) whether or not the destination's content is byte-identical. `contentHashAtCommit`, the now-unused `changedPathsBetween` wrapper, and the `basename` import were removed as dead code after the reduction. `tests/merge-gate.test.ts`'s R-C test was revised to cover what remains enforced: byte-identical archive move still passes and records matching hashes on the receipt; a contract renamed into the wrong family's archive shape (e.g. a contract landing on `tasks/archive/review-*.md`) still fails, since that is caught by the family-matched live-removal check, not by a destination hash. A known residual gap, honestly not covered: a SEMANTIC live path left completely untouched while a duplicate is separately added at its allowlisted archive-shaped destination would not be caught (the live path never appears in the F..L diff at all, so the removal check never runs on it) — see `tasks/todos.md`.
- **Fix 4 (P1, declaration parser hardening)**: `workflow_contract_evidence_requirement` (`assets/hooks/lib/workflow-state.sh:1533+`, projected via `bun scripts/sync-hook-sources.ts --write`) now counts every `evidence_requirements:` LINE across all yaml blocks (not merely how many blocks contain one) and fails closed unless exactly one exists anywhere in the file; within that one declaring block it computes a direct-child scope bounded by the first non-comment line at or above the declaring line's own indent (comments never end the scope or count as content) and requires `benchmark:` to occur exactly once inside that scope, failing closed on zero (including a `benchmark:` that exists only nested under an unrelated later sibling key) or on more than one (a duplicate key). Previously the parser took the first `benchmark:` line found anywhere after `evidence_requirements:` in the containing block, at any depth, silently accepting a coincidental deeply-nested value or the first of two contradictory declarations.

Recommendation returned `RECOMMENDATION: Defer destination-content binding to a dedicated immediately-following work-package, ship THIS PR now with a corrected/broadened residual-gap disclosure and an honest LOW severity re-rating — confidence: HIGH`, with reasoning the orchestrator accepted:
  - **Threat model**: the entire F→gate→archive→L sequence runs inside one local `contract-worktree.sh finish` process, one OS user, one worktree — no multi-party or network handoff exists in that window. The only actor who could inject bytes there already owns the worktree, the host-owned receipt store, and `git push`, and could bypass the gate directly; the gate defends against *drift* (reviewed F == merged L), not a compromised operator. The dangerous surface (production/test/config/asset paths) is already structurally foreclosed by fix-2's shape classifier, and the live semantic path is provably removed (not modified-in-place), so the contract's "any production, test, or contract semantic delta after freeze invalidates the receipt" already holds for every *live* surface. The unbound residue is non-executable, regenerated bookkeeping and dead archived copies, never re-read as gate/policy authority — an integrity-completeness gap, not an adversarial ship-authority breach.
  - **Critical fact that rules out rushing a fix**: `merge-gate.ts` explicitly refuses to run from the candidate repository (the reviewer must execute only trusted host code, never candidate code). `scripts/archive-workflow.sh` is candidate-repo code. A naive "have `merge-gate.ts run` invoke `archive-workflow.sh --predict`" would execute arbitrary candidate code *inside the trusted reviewer* — a worse hole than the one being closed. A correct fix requires the **driver** (`contract-worktree.sh finish`) to compute the prediction and hand it to `merge-gate.ts` as data, never let the reviewer execute candidate code itself. Bolting this on hastily in a fix-round risks getting exactly this wrong.
  - **Path ruled out**: shadow-parsing `archive-workflow.sh`'s transform logic a second time in `merge-gate.ts` (TypeScript) is forbidden by this repo's own one-parser-per-datum principle and would drift silently if the transform format ever changes.
  - **Corrected disclosure** (supersedes fix-round-1's narrower framing) recorded verbatim in `tasks/reviews/20260716-1419-closeout-authority-bootstrap.review.md` Residual Risks / Follow-ups.
  - **Follow-up work-package interface sketch** (decision-complete, for whoever opens it) recorded in `tasks/todos.md`, superseding the earlier vaguer `goal_sha256` deferral row: `archive-workflow.sh` gains `--predict-manifest` plus `--timestamp-human`/`--parent-run-id` overrides (same pattern as the existing `--timestamp`), runs the same transform logic against a scratch copy without mutating anything, emits `{"<dest-path>":"<sha256>"}`; `contract-worktree.sh finish_worktree` computes `finish_timestamp_human`/`finish_parent_run_id` once alongside the existing `finish_timestamp` and threads all three into both the predict call and the real archive step so predicted bytes match actual bytes; the driver calls predict *before* the real archive step and passes results to `merge-gate.ts run` as repeatable `--post-freeze-dest <path>=<sha256>` data flags (the reviewer never executes the candidate script); `merge-gate.ts run` validates each dest path against the SEMANTIC-dest shape and records it into `post_freeze_semantic`; `verifyReceipt`'s tolerance path restores content-hash comparison (`contentHashAtCommit`, removed as dead code in this PR's Fix 3) for every SEMANTIC dest path in the F..L diff, failing closed on mismatch or on a dest present in the diff but absent from the recorded manifest.

## Fix Round 7 — Ship-Attempt Flake Root Cause (2026-07-17)

Real `bash scripts/ship-worktrees.sh --ready` attempts intermittently failed at the contract-verification stage with `[FAIL] tests_pass: tests/helper-scripts.test.ts`, while the SAME test file passed cleanly and reliably in every standalone `bun test` run and in two independent clean reproductions of the full `verify-sprint.sh` chain (each confirmed via genuinely uncontaminated log capture — a non-invasive external `find`-based watcher that copies `run-bounded-verifier-command.ts`'s log files out of their `mktemp -d` temp directory as soon as the next sequential test's result file appears, so no source file was ever edited to obtain the capture). This apparent contradiction — same test, same file, sometimes clean/sometimes failing, never explained by subject staleness or system load alone — was root-caused by finally capturing the actual failing stderr from a genuine, unmodified real-ship attempt:

- **Root cause**: two named e2e fixtures inside `tests/helper-scripts.test.ts` — `"ship-worktrees default mode should finish without merging, push branch, and create draft PR"` and `"ship-worktrees should put dirty main closeout on a PR branch"` — each create a fake `gh` binary under a fixture-local `fakeBin` directory and prepend `fakeBin` to the child process's `PATH`, expecting `scripts/ship-worktrees.sh`'s own `gh_bin="${REPO_HARNESS_GH_BIN:-gh}"` resolution to pick up the fake via bare-`gh` PATH lookup. Unlike the sibling `REPO_HARNESS_BASH_BIN`/`REPO_HARNESS_BUN_BIN`/`REPO_HARNESS_GIT_BIN` overrides already present in the same env objects, neither fixture set `REPO_HARNESS_GH_BIN` explicitly — the ONLY one of the four trusted binaries relying on bare PATH-based resolution instead of an absolute-path override. Under specific conditions this bare-PATH lookup fell through to the real system `/opt/homebrew/bin/gh` instead of the fixture's fake, and the real `gh pr create` then genuinely failed (`gh auth login` / `GH_TOKEN` guidance in stderr) against the fixture's fake local git remote, which has no real GitHub-hosted repository behind it.
- **Why intermittent, not deterministic**: bare-command PATH resolution inside a deeply nested spawn chain (bash → bash → bash, several levels of `run_cmd`/`spawnSync`) is exactly the kind of environment-propagation edge case that is sensitive to process/shell state and does not reproduce reliably in isolation — consistent with every other trusted binary in this same env object already having been given an explicit override precisely to avoid this class of fragility, while `gh` was the one left out.
- **Fix**: added `REPO_HARNESS_GH_BIN: join(fakeBin, "gh")` to both fixtures' env objects, matching the existing explicit-absolute-path pattern used for the other three trusted binaries. Verified both tests individually post-fix: the "default mode" test dropped from consistently reaching the slow real-`gh`-auth-check path (60-220+ seconds observed across many real-ship attempts) to 18.79s; the "dirty main closeout" test dropped to 1.78s. Both are now deterministic (the fake `gh` script itself is instant, no network round-trip).
- **Verified via genuinely clean reproduction, not source-editing diagnostics**: the fix was confirmed by directly reading the real stderr captured from an actual, completely unmodified `bash scripts/ship-worktrees.sh --ready` invocation via the external non-invasive watcher described above — this is the first time in this investigation the real failure content was observed without the diagnostic method itself contaminating the result (every earlier attempt that edited `scripts/verify-contract.sh`, even trivially, self-triggered the unrelated `content drift` or `subject staleness` checks and obscured the real signal).
- No product/design code was touched — this was a test-fixture-only reliability gap in two pre-existing e2e fixtures, not a defect in this PR's own `contract-worktree.sh`/`merge-gate.ts`/`workflow-state.sh` changes.
