## Design Decisions

### Phase A1 — apply_patch full-batch scope

- Stop condition check (recursive payload losing the original patch command): confirmed **not triggered**. `expanded_payload` at `assets/hooks/pre-edit-guard.sh:28` (unchanged by this fix) already embeds `tool_input.command` alongside `tool_input.file_path` on every recursive expansion, so `resolve_edit_workflow_profile()` can always re-derive the full path set via `hook_get_apply_patch_paths` inside the same process (stdin is cached per-process by `hook_read_stdin_once`, so re-reading `.tool_input.command` there is free). Kept a defensive fallback to `$FILE_PATH` if re-extraction ever yields zero paths, so the code never crashes even though this branch is structurally unreachable today.
- `--profile` is added to the `args` array before the variadic `--target-path "${target_paths[@]}"` (which is now always last), per the dispatch's explicit ordering constraint, to avoid any risk of commander's variadic consumption swallowing a later flag.
- Non-patch (plain Edit/Write) calls are untouched: `target_paths` only gets the batch treatment when `REPO_HARNESS_APPLY_PATCH_PATH_EXPANDED=1` **and** `.tool_input.command` is non-empty; otherwise it falls back to the single `$FILE_PATH`, exactly as before.
- Test placement: the six batch-scope integration cases went into `tests/runtime-profile-enforcement.test.ts` as instructed. Test (e) (strict-category leak) had to order the plain implementation path *before* the strict-category path in the patch text — `is_private_ops_path`/profile-resolution ordering means the recursion checks paths in patch order and exits on the first failure, so to prove the *sibling* gets promoted (not just the strict file itself, which was already strict pre-fix), the sibling must be checked first. The assertion pins the exact `StrictContractGuard` message text (`Strict workflow edit to src/plain1.ts has no active contract.`), which only appears post-fix.
- **Found and fixed a real fallout in a pre-existing test** (not one of the six new cases): `Codex apply_patch expands every target path and blocks high-risk or private writes` (existing test, same file) patched `src/safe.ts` before `_ops/secret.txt`. Post-A1, the batch-wide strict-token scan sees `_ops/secret.txt`'s "secret" token and promotes `src/safe.ts` to strict too, so it now fails on `SpecGuard` (no `docs/spec.md` in the fixture) *before* the loop ever reaches `_ops/secret.txt`'s own dedicated `OpsPrivateGuard` check. This is the intended, plan-documented consequence of A1 (risk #2, "A1 over-promotion": raise-only, never lowers). Fixed by reordering that patch so `_ops/secret.txt` is checked first — `is_private_ops_path` is a pure path match evaluated before profile resolution, so it now fires independent of batch-scope ordering effects. The write is still blocked (exit 2) either way; only the guard name that fires first changed.
- TDD evidence: pre-fix run of the six new cases (`.ai/harness/runs/phase2-a1-prefix-failure.txt`, `PRE_FIX_EXIT=1`) shows b/c/d/e/f failing (5 fail, 6 pass) and (a) already passing — matches the plan's live-repro description exactly (3-file batches were never broken; 4-file / cross-capability / strict-leak batches were).

### Phase A2 — single-field `state resolve` output

- Confirmed `assets/hooks/post-edit-guard.sh` has **no** `state resolve` call and no `workflow_profile` parsing at all (`grep` for both terms returns zero matches) — so A2 only touches `pre-edit-guard.sh` and `src/cli/commands/state.ts`, not `post-edit-guard.sh`.
- Confirmed the only consumer of `resolve_edit_workflow_profile()`'s stdout is the single assignment `WORKFLOW_PROFILE="$(resolve_edit_workflow_profile || true)"` — no other field from the JSON is read by the guard — so `--field workflow_profile` alone is sufficient; no other fields needed projecting.
- `--field <name>` is a pure output projection on `state resolve --json` (resolver untouched): prints the raw string for string fields (matching the old `jq -r '.workflow_profile // empty'` behavior byte-for-byte, including printing nothing for `null`/`undefined`), or `JSON.stringify` for non-string fields (covered by a test using `blockers`, an array). An unknown field name prints nothing and still resolves + persists state underneath (also covered by a test) — it does not error, matching the fail-open-on-output/fail-closed-on-exit-code posture of the rest of this command (exit code is still `blockers.length > 0 ? 1 : 0` regardless of `--field`).
- With `--field workflow_profile` in place, the old `if command -v jq ... elif command -v bun -e ...` block in `pre-edit-guard.sh` is dead code and was deleted outright (not kept as a fallback) — the CLI now always returns exactly the projected value in one call, so there is nothing left to parse.
- Test placement: `tests/cli/state-snapshot.test.ts` actually tests the *separate* `state-snapshot` (singular) hook-entry command via `hook-entry.ts`, not `state resolve`. `tests/effective-state.test.ts` is the file that already exercises `state resolve --json` through the real CLI (`runCli` helper, previously unused/dead code — now exercised by the new tests), so the four new `--field` cases went there instead, per the dispatch's "or a nearby test file" allowance.
- Perf verification caveat: `jq` is installed on this dev host (`/usr/bin/jq`, jq-1.7.1), so the pre-fix fallback (`bun -e`) was never actually exercised here — the pre-fix second-parse step was already the fast `jq -r` path, not a second Bun cold start. A direct 5-run timing comparison of `pre-edit-guard.sh` on a single plain edit (stash-compared before/after) shows no measurable difference on this host (~628ms both ways — dominated by the first, unavoidable `bun state resolve` cold start in both versions). The A2 win is structural and host-independent: exactly one CLI invocation and zero fallback-parse subprocess in every case, verified by reading the diff (the `bun -e` call site is gone, not conditionally skipped) — the previous 1-or-2-process shape collapses to always-1. The full `bun scripts/hook-dispatch-diet-report.ts` p95 gate independently fails on its `state-snapshot` probe (a different command, `hook-entry.ts state-snapshot`, never touched by A1/A2); stash-verified this reproduces identically (615ms) — actually slightly worse — on unmodified code, so it is a pre-existing environmental condition on this worktree (fresh `node_modules`, cold Bun/OS cache), not a regression from this change. `script_invocation_count` (15) is unaffected either way — it counts hook script *files* wired into `route-registry.ts`, none of which A1/A2 added, removed, or renamed.
- `node_modules` did not exist in this fresh worktree at task start (`bun install` had never been run here); ran `bun install` to unblock `bun test` and the other required-check commands. Unrelated to A1/A2 but necessary to produce real verification evidence.

### Falsifier check (B1 landed → before B2 code)

Contract falsifier: B2's premise ("Lite's task artifacts are agent-authored `plans|tasks` ceremony, not hook spill under `.ai/harness/runs|checks`") is falsified if B1 shows the artifacts actually live under `.ai/harness/runs|checks`.

**Structural proof (decisive, not just this run's data):** `.ai/harness/runs/` and `.ai/harness/checks/*.latest.{json,md}` are gitignored (`.gitignore:36-39,56`; verified in both the repo root and a live benchmark workspace's copy of `.gitignore`). `changedFiles()` in the benchmark script (`scripts/run-harness-profile-benchmark.ts:357`) is exactly `git status --porcelain=v1 -uall`, **without** `--ignored`. `-uall` only expands untracked-directory recursion depth; it does not surface gitignored paths. Confirmed empirically: inside a leftover Lite benchmark workspace, `git status --porcelain -uall` shows zero `.ai/harness` entries, while `git status --ignored -uall` on the *same* workspace shows 10+ real hook-written files under `.ai/harness/runs/*.json` and `.ai/harness/checks/*.json` that exist on disk but are invisible to the first command. **This means `artifact_files_created`/`artifact_files` structurally cannot count hook-spilled `.ai/harness/runs|checks` files** — the contingency branch's premise isn't just unproven today, it's unreachable by the current counting method, gitignored or not. (Tracked `.ai/harness/*` files like `policy.json`/`workflow-contract.json` could in principle show as `M` if a task modified them, but none of the scenarios touch those.)

**Direct evidence (this run, not the plan's cited `36252e77`/`8237454` baseline — a different, older/leftover run whose tmp workspace under `$TMPDIR/repo-harness-profile-benchmark-saNynf` survived on this shared-host machine; the benchmark script never cleans up its tmp root):** ran `git status --porcelain -uall` inside all four surviving `{adaptive-lite,strict-harness} x {cross-capability-feature,database-migration}` workspaces.
- `adaptive-lite/cross-capability-feature` (artifacts=4): `plans/plan-20260713-0925-shared-status-format.md`, `tasks/contracts/20260713-0925-shared-status-format.contract.md`, `tasks/notes/...notes.md`, `tasks/reviews/...review.md` — all 4 under `plans/`/`tasks/`, freshly dated, zero under `.ai/harness/`.
- `adaptive-lite/database-migration` (artifacts=5): `tasks/todos.md` + `plans/plan-20260713-0933-add-widget-status.md` + `tasks/contracts/...`/`notes/...`/`reviews/...` — all 5 under `plans/`/`tasks/`, zero under `.ai/harness/`.
- `strict-harness/cross-capability-feature` (artifacts=4): the pre-seeded `plans/plan-20000101-0000-benchmark.md` + `tasks/contracts/20000101-0000-benchmark.contract.md` (Strict's 2-file projected envelope, modified not created) plus agent-added `tasks/notes/...`/`tasks/reviews/...` completing that *same* seeded slug — structurally different from Lite, which authors a brand-new dated slug unprompted.
- `strict-harness/database-migration` (artifacts=0): only `deploy/sql/...` changed, which doesn't match the counting regex.

Every matched path across all four workspaces is under `plans/` or `tasks/`; none under `.ai/harness/`. Conclusion: **existing data (plus the structural gitignore proof) supports the `plans|tasks` ceremony hypothesis** — falsifier not triggered. Per the contract's first branch, proceeding to B2's main design. Caveat recorded honestly: this leftover run's aggregate counts (Lite 4+5=9, Strict 4+0=4) don't match the plan's cited baseline table (Lite 5+5=10, Strict 0+0=0) exactly — it's a different historical run on the same shared-tmp host, not the run the plan's Evidence Baseline cites. The end-of-B live rerun (fresh, on this exact code) is the actual authoritative confirmation regardless; this check only had to clear the "don't build B2 on a falsified premise" bar before writing any B2 code, which it does independent of which specific historical run is inspected, because the gitignore argument holds for any run.

### Phase B2 — ceremony trigger threshold

- **Field design: new `guidance` field, not an overload of `next_action`.** The dispatch named `next_action`/`guidance` as candidate slots. `next_action` already has an established, narrower semantics ("the literal next TODO-list step": `'resolve blockers'`, the first open plan-task bullet, or handoff's exact-next-step) with two exact-match assertions in `tests/effective-state.test.ts` (`'implement resolver'`, `'resolve blockers'`). Overloading it with ceremony text would conflate two different guidance categories and break those assertions for no benefit. Added `guidance: string | null` alongside it instead — additive, zero collateral changes to `next_action` consumers.
- **`CEREMONY_GUIDANCE` lives in `state-snapshot.ts`, keyed by the existing `WorkflowProfile` union** (`lite`/`standard`/`strict`), computed as `riskResolution.ok ? CEREMONY_GUIDANCE[riskResolution.profile] : null` — mirrors the existing `workflow_profile: riskResolution.ok ? riskResolution.profile : null` null-on-blocked pattern exactly. `resolveWorkflowProfile()` itself (the floor) is untouched, per the plan's explicit prohibition.
- **`runtime.ts` needed a wiring change even though the plan's B2 file list didn't name it.** `effectiveStateSessionSection`'s `compact` object is a hand-picked whitelist of `EffectiveState` fields serialized into the `[HarnessState]` JSON blob (`runtime.ts:158-175`); a new top-level field on `EffectiveState` does not automatically reach the SessionStart output unless also added to this whitelist. The dispatch explicitly pointed at this exact function/line-range ("看 `effectiveStateSessionSection` 怎么消费"), which only makes sense if a change there was expected; without it, `guidance` would exist in `repo-harness state resolve --json` output but never reach the agent at session start, defeating B2's actual goal. `src/` is unscoped in the contract's `allowed_paths` (no narrower subpath restriction), so this is in-bounds. Two-line change: added `guidance?: unknown` to the parsed-JSON shape and `guidance: state.guidance ?? null` to `compact`, following the exact pattern already used for `next_action` one line above each.
- **Did not add `guidance` to `session-context-budget.ts`'s `CRITICAL_STATE_FIELDS`.** That list feeds `criticalFieldStats`, used only inside `mandatoryOverflow()` to report which safety/identity fields (task_id, phase, next_action, blockers, forbidden_paths/actions, security/safety boundaries) survive when a mandatory section is replaced by the fail-closed overflow payload. `guidance` is advisory text, not a safety-gate field — matches the dispatch's own "guidance 是文本层，不是新 gate" boundary. Not adding it is the more conservative, more clearly in-scope choice; the field still fully serializes into the JSON content either way since `criticalFieldStats` doesn't filter `compact`, it only adds separate diagnostic stats on overflow.
- **`assets/hooks/session-start-context.sh`: no change made.** Read the entire file (573 lines) and grepped for any existing profile-conditioned ceremony text (plan/contract/notes/todos/checks authorship instructions gated on task size). Found none — every section in this file (`Pending Plan Capture`, `Current Status Snapshot`, `Active Sprint`, `Capability Context Queue`, `Architecture Queue`, `Tooling Update Advisory`) is conditioned on an explicit, unrelated signal (a pending host planning discussion, an active sprint backlog item, a queued capability/architecture drift request, a stale tooling report) — none of them is a blanket "author ceremony for every task" instruction that could contradict Lite's zero-ceremony rule. The dispatch's own phrasing was conditional ("如有...改为渲染...不留第二份拷贝"); since there is no existing copy, there is nothing to de-duplicate. Deliberately did **not** add a new `state resolve` shell-out here to render `guidance` a second time: `effectiveStateSessionSection` in `runtime.ts` already runs once per `SessionStart` *before* any route script (including this one) executes, and pushes the `[HarnessState]` blob (now carrying `guidance`) as a `priority: 2, mandatory: true` section — the highest-priority, budget-survival-guaranteed channel in `budgetSessionContext`. A second CLI invocation from this script would duplicate a Bun cold start on every session start, directly undercutting Phase A2's cold-start-reduction goal, for content that's already delivered through a channel with *better* delivery guarantees than this script's own `priority: 5` sections. `.ai/hooks/*` therefore needed no re-projection; verified with `bun run sync:hooks` (no-op, same content hash) and `bun run check:hooks` (clean, 25 files).
- Test placement: added the three-tier assertions to `tests/effective-state.test.ts` (named first in the dispatch's "or a nearby file" instruction), reusing the existing `withRepo`/`resolveFixtureState` fixture. Had to add a `setContractProfile()` helper doing a regex replace of the whole `> **Workflow Profile**: ...` header line instead of chaining `.replace('standard', 'lite')` then `.replace('standard', 'strict')` — the second chained replace silently no-ops once the first replace has already removed the literal substring `'standard'` from the file, which produced a `PROFILE_BELOW_RISK_FLOOR` (`workflow_profile: null`) failure on the first attempt (explicit override stayed `'lite'` while `resolveFixtureState`'s `operationKind: 'feature'` computed riskFloor `'standard'`). `tests/hook-runtime.test.ts` was not touched: its own `runHook()` test helper spawns each hook shell script directly and in isolation (never through `runtime.ts`'s real `runHook`/`effectiveStateSessionSection`/`budgetSessionContext` pipeline), so nothing in that file exercises the `[HarnessState]` JSON shape I changed — confirmed by grep (no `HarnessState`/`effective-state` hits) before deciding not to touch it. `tests/cli/state-snapshot.test.ts` also not touched: it tests the unrelated legacy `state-snapshot` (singular) hook-entry command (`StateSnapshot`/`SnapshotPlanState` types), not `state resolve`/`EffectiveState` at all.
- Verification for B2: `bun test` (full suite, 1249 pass / 1 skip / 0 fail), `bun run check:type` (clean), `bun run sync:hooks` + `check:hooks` (no-op/clean), plus the plan's repo-wide required checks (deploy-sql-order, architecture-sync, task-sync, check-task-workflow --strict, inspect-project-state, adopt --dry-run) — all exit 0.

## Tradeoffs Considered

| Option | Decision | Reason |
|--------|----------|--------|
| Keep `jq`/`bun -e` fallback in `resolve_edit_workflow_profile()` for defense-in-depth | Rejected — deleted outright | `--field` moves the projection into the CLI itself; the shell-side parse has nothing left to do. Keeping dead branches contradicts the plan's minimal-change / no-steady-state-compatibility-code policy. |
| Broaden the reordered `_ops/secret.txt` test assertion to a regex instead of reordering the patch | Rejected — reordered the patch instead | Reordering preserves the original test's specific intent (OpsPrivateGuard still catches private-ops writes in a batch) and is order-independent of any future profile-resolution change, since `is_private_ops_path` never depends on `WORKFLOW_PROFILE`. |
| Put A2's `--field` tests in `tests/cli/state-snapshot.test.ts` (named in the dispatch) | Rejected — used `tests/effective-state.test.ts` instead | The named file tests the unrelated `state-snapshot` (singular) command; `effective-state.test.ts` already exercises `state resolve --json` through the CLI and is the dispatch's allowed "nearby test file" alternative. |
| B2: append ceremony text onto the existing `next_action` string | Rejected — added a new `guidance` field instead | `next_action` has an established, narrower meaning with exact-match test coverage; conflating it with ceremony rules breaks that contract for no functional gain over an additive field. |
| B2: render `guidance` by shelling out to `state resolve --json` a second time from `session-start-context.sh` | Rejected — left the file unchanged | The TS-side `effectiveStateSessionSection` already delivers `guidance` via a higher-priority (`mandatory: true`, priority 2) section before any route script runs; a second CLI call would add a duplicate Bun cold start to every SessionStart, working against Phase A2's goal, for content already delivered through a better-guaranteed channel. |
| B2: add `guidance` to `session-context-budget.ts`'s `CRITICAL_STATE_FIELDS` | Rejected — left the list unchanged | That list is for safety/identity-gate field stats surfaced on mandatory-section overflow; `guidance` is advisory text, not a gate field, and it still fully serializes into the JSON content regardless (the list doesn't filter output, only adds diagnostic stats). |
