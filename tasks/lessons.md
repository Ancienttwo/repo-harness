## Active Lessons

- Date: 2026-07-11
- Triggered by correction: external ship review showed that `install-agent-fleet.sh` granted `workspace-write` from the parsed frontmatter `name`, so a mismatched `gatekeeper.md` declaring `name: fast-worker` crossed the intended privilege boundary.
- Mistake pattern: deriving permissions from self-declared source metadata without first binding that metadata to the trusted managed identity selected by the installer loop.
- Prevention rule: validate filename/registry identity against semantically normalized, structure-aware metadata before any write. Require the declared flat YAML root schema rather than inferring from nested fields, and reject duplicate root keys because an ambiguous parse is never deletion authority; use the authoritative runtime parser for TOML and compare parsed identity literally to the known role set. Apply that exact-known-role restriction to every format so YAML non-string scalars remain ambiguous rather than deletion authority. Text scans and cross-format reinterpretation are not deletion authority. Enforce the parser runtime's real minimum version in both platform bootstrap scripts and the shared CLI boundary reached by every public mutating install/update path; package-engine metadata may be ignored. When multiple runtime locations are supported, test every trusted candidate before failing so a stale PATH entry cannot mask a compatible explicit fallback. Bind later subprocesses to the exact executable that passed validation, and self-upgrade only a runtime owned by its own installer; package-manager-owned runtimes must fail closed with the manager's upgrade command. Missing or unparseable source identity is fatal; derive sandbox privileges only from validated trusted identity. Inspect installed identities before ordinary drift handling so corrected sources repair stale privilege-bearing targets, and base success on the resulting fleet state rather than only on whether remediation happened in the current process. Unresolved recovery must stay non-zero across retries while ambiguous or identity-correct user drift remains untouched.
- Where to apply next time: installers, manifest projections, role registries, and any generator that maps user- or upstream-authored metadata into runtime permissions.

- Date: 2026-07-06
- Triggered by correction: authority-closure T2 — adding one apostrophe ("task's") to `PI_TEMPLATE_CONTRACT` broke `scripts/lib/project-init-lib.sh` at `bash -n` (exit 2, "unexpected EOF while looking for matching") on stock macOS bash 3.2.57.
- Mistake pattern: a `$(cat <<'EOF' ... EOF)`-wrapped heredoc with a QUOTED delimiter still gets its body scanned for quote parity by bash 3.2's `$()` boundary matcher; an odd count of single quotes inside the literal body breaks parsing even though the heredoc content is never interpreted. Minimal 5-line repro confirmed; other `PI_TEMPLATE_*` heredocs had even quote counts by luck.
- Prevention rule: never embed prose with apostrophes inside `$(cat <<'EOF')` on scripts that must run under /bin/bash 3.2; use temp-file indirection (`mktemp` + `cat > file <<'EOF'` + read back) as `project-init-lib.sh` now does. `read -r -d ''` is not a substitute here — it exits 1 at heredoc EOF and every consumer of this lib runs `set -euo pipefail`. Always `bash -n` every template-bearing .sh after editing embedded template text.
- Where to apply next time: any slice that inserts new template/field prose into `scripts/lib/project-init-lib.sh`, `scripts/ensure-task-workflow.sh`, or `scripts/plan-to-todo.sh` heredocs; check quoting mode per heredoc before writing text containing `'`, backticks, or `$`.

- Date: 2026-07-03
- Triggered by correction: a full `bun test` run leaked a real `plans/sprints/20260703-1424-auth-overhaul.sprint.md` + `.ai/harness/sprint/active-sprint` marker into this repo; content matched `tests/sprint-backlog.test.ts:108`'s fixture exactly.
- Mistake pattern: shell-spawning test helpers that pass `env: undefined` (or `env ? {...} : undefined`) to `spawnSync` silently inherit the parent bun-test process's full `process.env`. Since `bunfig.toml` sets `maxConcurrency = 4`, all test files share one process; a leaked/poisoned env var (e.g. `REPO_HARNESS_TARGET_REPO_ROOT`, which `scripts/sprint-backlog.sh`'s WIP cwd-resolution branch trusts blindly) from anywhere in that shared process can silently redirect a "sandboxed" tmp-workspace test into the real repo. Root cause NOT confirmed — isolated single-file and 7-file/172-test concurrent reruns both failed to reproduce; likely needs full 93-file suite load or was a one-off timing fluke. Applied `tests/sprint-backlog.test.ts`'s explicit `SANDBOX_ENV_BLOCKLIST` strip as defense-in-depth regardless.
- Prevention rule: any test helper that spawns a subprocess into a tmp sandbox must construct an explicit `env` object and delete known ambient-leak vars (`REPO_HARNESS_TARGET_REPO_ROOT` and friends) rather than passing `env: undefined`/raw `process.env` through. Product helpers that accept target-root env must also fail closed unless the canonical env root matches the helper's current cwd; do not rely on tests to scrub inherited env. Never trust "isolated run didn't leak" as proof of absence in a shared-process, concurrent test runner.
- Where to apply next time: audit other test files under `tests/` that spawn `scripts/*.sh` copies into `mkdtempSync` workspaces (grep `spawnSync.*cwd` alongside `env: undefined` or omitted `env`) for the same blind-inheritance pattern; revisit once `scripts/sprint-backlog.sh`'s WIP `REPO_HARNESS_TARGET_REPO_ROOT` change lands and re-attempt a full-suite repro with the fix reverted to confirm/deny the exact trigger.

- Date: 2026-07-06
- Triggered by correction: self-caught while writing a new `tests/cli/global-runtime-init.test.ts` spawnSync test for the `install`/`init` optional-dep interactive-prompt change; the new test hung/produced empty stdout+stderr with exit 0 and no thrown error.
- Mistake pattern: calling `spawnSync('bun', [CLI, ...], { env: { ...process.env, PATH: \`${fakeBin}:${process.env.PATH}\` } })` when the test also drops a fake `bun`/`bunx` shim into `fakeBin` for intercepting the CLI's *internal* subprocess calls. A literal `'bun'` command string plus a manual bash-level PATH edit shadows the real bun runtime too — but empirically `child_process.spawnSync` does NOT use `options.env.PATH` to resolve the command executable itself (only `process.env.PATH` of the actual host process), so this specific failure mode does not reproduce via `spawnSync` — the actual bug was elsewhere (docs. below) and cost real debugging time via a misleading manual-bash repro that used shell-level PATH lookup instead of spawnSync's own resolution. [Corrected 2026-07-09, isolated probes under both runtimes: the "does NOT use options.env.PATH" claim only holds under Bun when NO `env` option is passed at all — Bun then resolves via the PATH the OS process started with, ignoring live `process.env.PATH` JS mutations. When `options.env` IS passed, Bun's `spawnSync` DOES resolve the executable via `options.env.PATH` (an injected fake-bin PATH intercepts the command even though live PATH never changed). Node diverges in the omitted-`env` case: it picks up live `process.env.PATH` mutations. So state the branch explicitly before reasoning about spawn resolution: (runtime: bun|node) × (env passed|omitted).]
- Prevention rule: when debugging a `spawnSync`-based CLI test, reproduce with a small throwaway script that calls `spawnSync` directly (inspecting `res.status`/`res.error`/`res.stdout`/`res.stderr`) instead of hand-rolling the equivalent as a bash one-liner with shell PATH overrides — the two are not equivalent and the bash version can send you chasing a phantom shadowing bug. Separately: this repo's existing convention (see `tests/cli/global-runtime-init.test.ts`'s `CLI update refreshes user-level runtime...` test) is to spawn with `process.execPath`, not literal `'bun'`, whenever the test's `env.PATH` is overridden with a fake bin dir; keep using `process.execPath` for consistency even though the literal-`'bun'` shadowing theory itself didn't hold up.
- Where to apply next time: any new `tests/cli/*.test.ts` spawnSync-based CLI smoke test that overrides `PATH` with a fake bin directory; prefer `process.execPath` as the spawn command, and if output is unexpectedly empty, inspect `res.status`/`res.error`/`res.signal` directly before assuming a PATH-resolution shadowing bug.

- Date: 2026-07-09
- Triggered by correction: user report — on a machine with repo-harness 0.8.3 previously installed, `bunx repo-harness install` kept installing 0.8.3 instead of the latest release.
- Mistake pattern: two self-reinforcing layers. (1) `bunx`/`npx` without an explicit `@latest` reuse the already-installed global binary or cache copy and never consult the registry, and README recommended the unpinned form. (2) The running copy's `installCli` step (src/cli/commands/global-runtime.ts) either skips the reinstall when the running copy IS the Bun global package, or installs its own `sourceRoot` copy globally — every path re-installs the version that is already running, so a stale version silently self-perpetuates.
- Prevention rule: any doc or script that bootstraps a CLI via `bunx`/`npx` must pin `@latest` (or an explicit version); an "already installed, skipping" fast path may surface a best-effort registry comparison only behind the explicit `REPO_HARNESS_CHECK_UPDATES=1` opt-in. The lookup must use the running Bun executable, a fixed official registry, and a non-repo cwd so an untrusted checkout cannot supply the package-manager binary or registry config. The upgrade path (`repo-harness update` → `repo-harness@latest`) existed but nothing pointed users at it.
- Where to apply next time: README install snippets, install.sh/install.ps1, and every `installCli`-like self-bootstrap branch that can conclude "already installed" from a potentially stale running copy.
