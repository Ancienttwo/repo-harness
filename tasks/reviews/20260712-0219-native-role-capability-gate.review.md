## Third Round Verification (2026-07-13)

- A third independent Codex review of the second-round mkdir lock found it still had a real gap: `acquireLock`'s stale-reclaim only checked `mtime > 8s`, with no check of whether the current holder was actually still alive, and `releaseLock` unconditionally `rmdirSync`d the lock directory with no ownership check. Concrete failure: holder A pauses (GC, system load, a slow subprocess) past the 8s stale threshold while still legitimately inside its critical section (not crashed); waiter B sees the stale mtime and reclaims by deleting A's lock directory, then proceeds under its own (correct, from B's perspective) critical section; A eventually resumes, still believes (via its own local return value) that it holds the lock, clobbers whatever B legitimately wrote, then calls the old unconditional `releaseLock`, which deletes B's lock directory (or a third process C's, if C has since acquired) regardless of current ownership. This breaks the mutual exclusion the lock exists to provide.
- Fixed with an ownership token rather than narrowing the mtime window further: `acquireLock` now writes an `owner.json` marker (`{ pid, token, acquired_at }`) inside the lock directory immediately after `mkdirSync` succeeds, and returns `{ acquired, token }` instead of a bare boolean; both call sites (`subagent-start-context.sh`'s `latest.json` read-compare-write and `codex-delegation-advisor.sh`'s `latest.json` write) were updated to check `.acquired` and pass `.token` through to `releaseLock`. `releaseLock(lockDir, token)` now reads the current `owner.json` first and only removes the lock when the token still matches; a mismatch or missing marker means a different holder now owns it (or it is already gone), so the call leaves it untouched instead of blindly deleting. Stale-reclaim now additionally calls a new `isLockOwnerAlive(lockDir)` helper before reclaiming, which reads `owner.json` and probes `process.kill(pid, 0)`: a clean call or `ESRCH` means the owner is gone (safe to reclaim), `EPERM` means it is still alive but not signalable by this process (do not reclaim), and a missing or corrupt marker is treated as an abandoned lock from a crash in the brief mkdir-to-marker-write window (also safe to reclaim once stale). Reclaim now requires BOTH the mtime-stale condition AND a confirmed-dead (or marker-less) owner, matching the dispatch's explicit requirement; the ~2s bounded acquire retry and skip-not-hang-on-timeout behavior are unchanged, and no new dependency was added.
- Two new regression tests added to `tests/cli/hook.test.ts`, this time exercising `acquireLock`/`releaseLock` directly (extracted verbatim from the real `assets/hooks/subagent-start-context.sh` via a small brace-counting harness builder, then run as genuine separate OS processes) rather than through the full hook CLI route, so `staleMs` could be lowered for a fast, deterministic test instead of sleeping past the real 8s default:
  - `acquireLock does not let a concurrent attempt steal the lock from a still-live holder past the stale threshold` — holder A acquires and holds via a real synchronous sleep (genuinely live, not a stale artifact) for 1000ms; the test waits `staleMs(150)+100ms` so the lock's mtime has actually aged past the stale threshold before contender B's bounded 300ms attempt (comfortably inside A's remaining hold window); asserts B never acquires, and that A's own release still succeeds once it legitimately finishes.

## Fourth Round Verification (2026-07-13)

- A fourth independent Codex review of the third-round mkdir-plus-owner.json lock found two more real races, and per explicit user decision this round replaces the mechanism outright rather than patching it a fourth time (three consecutive rounds each finding a new TOCTOU gap in the same non-atomic multi-step primitive is itself the signal that the primitive was the defect):
  1. `mkdirSync(lockDir)` succeeding and the separate `owner.json` write were not atomic together. A holder pausing in exactly that gap for long enough could look stale-and-markerless to a waiter, which reclaims (deletes and recreates); the original holder's own marker write (never checking whether its directory had been replaced) then landed inside the reclaimer's new directory, overwriting its marker, so both believed `acquired:true` and could enter the critical section simultaneously.
  2. The stale-reclaim path itself (check owner is dead, then delete, then mkdir again) was its own check-then-act sequence: two concurrent reclaimers could race so one deletes the lock the other had just legitimately recreated.
- Replaced the directory-plus-separate-marker-file design with a single lock FILE per `${latestPath}.lock` (the `.lock` directory convention is gone), acquired via `fs.writeFileSync(lockPath, JSON.stringify({ pid, token, acquired_at }), { flag: "wx" })`. `wx` is `O_CREAT | O_EXCL | O_WRONLY`: one call atomically creates and opens the path (throwing `EEXIST` synchronously if it already exists) and writes the small owner-marker payload as part of that same successful open, so there is no intermediate state where the lock exists but has no readable owner yet — this closes race (1) structurally, not by narrowing the window further. This assumes the marker payload stays small and the hook always runs against a local or tmpfs-backed filesystem (documented explicitly in the `tryCreateLockFile` comment); it does not attempt to solve the general case of multi-writer torn writes for a larger payload or a networked filesystem.
- Stale-reclaim keeps the same shape (mtime-stale AND confirmed-dead-or-unreadable owner via `isLockOwnerAlive`, adapted to read the lock file directly instead of an `owner.json` inside a directory). Closing race (2) took **two attempts within this same round**, and both are recorded here honestly rather than presenting only the one that worked, because the first attempt is itself an instructive finding:
  - **First attempt (insufficient):** on a reclaimable lock, capture a byte-for-byte content snapshot at the moment of the `reclaimable` decision, and immediately adjacent to the destructive `unlinkSync` call re-read the path and require the content to still match that snapshot before unlinking. Building the required concurrent-reclaimers regression test (below) first reproduced the underlying bug empirically against a content-blind `unlinkSync` (`Received: 2`, both racers believing `acquired:true`, on the very first trial), then this content-verification fix was applied — an initial 5-run sample looked clean (5/5 pass), but a rigorous 15-run follow-up immediately failed 3/15 with the identical `Received: 2` output. Root cause of the fix itself failing: re-reading once more before acting is still a check-then-act sequence, not an exclusive operation — two racers running the identical code from the identical forced starting point routinely complete that re-read while the original content is still unchanged for BOTH of them, before either has acted, so both still proceed to the destructive step together. This is the same shape of mistake the whole round exists to correct (narrowing a non-atomic window instead of replacing it with a real atomic primitive), now discovered recurring one level down, inside a fix applied within this very round — a genuinely useful, self-correcting finding surfaced only by testing at real volume (15+ runs) rather than trusting a single small clean sample.
  - **Actual fix:** gate entry into the destructive unlink-then-recreate step behind its own exclusive marker file, `${lockPath}.reclaiming`, created with the identical `wx` primitive as the main lock. Only the process whose own `tryCreateLockFile` call for the marker returns `created:true` proceeds to unlink and retry-create the main lock, inside a `try`/`finally` that releases the marker afterward via the existing token-verified `releaseLock` (not a raw unlink), regardless of whether the main-lock reclaim itself succeeded. A process that loses the marker create does not touch the main lock on that pass at all; it only reclaims the marker itself if the marker looks mtime-stale AND confirmed-dead by the same two-part test as the main lock — accepted as an extremely low-probability path, since the marker is only ever held for a handful of synchronous, non-sleeping syscalls, so a genuinely abandoned marker implies a hard kill inside that tiny window. This closes the race by demoting "two reclaimers racing the same destructive step" to an ordinary contended `wx` create, which the OS already guarantees has exactly one winner — a real exclusivity guarantee, not a narrower probabilistic check.
- The single correctness property for ownership, traced through every return path in `acquireLock`: a call believes `acquired:true` if and only if its OWN `tryCreateLockFile` invocation (the initial attempt, or the reclaim retry reached only after independently winning the reclaim marker) returned `created:true` — never from having merely performed a preceding unlink, and never from having won the reclaim marker alone.
  - `if (attempt.created) return { acquired: true, token: attempt.token };` — gated directly on this call's own initial create.
  - `if (reclaimAttempt.created) return { acquired: true, token: reclaimAttempt.token };` — reached only inside the `else` branch where `reclaimMarker.created` is true (this call's own marker-create succeeded), but still gated on its own separate `tryCreateLockFile` call for the MAIN lock, never on the marker win by itself; the marker only earns the right to attempt this, it does not grant ownership.
  - Every other path (`catch` blocks, the marker-loser branch, the post-reclaim fallthrough, and the deadline check) returns `{ acquired: false, token: null }` or loops; none of them claim ownership.
- Release (`releaseLock`) keeps the same read-verify-then-unlink shape as the third round, adapted to read the lock file directly: only unlinks when the current content's token still matches the token this exact `acquireLock` call was given back.
