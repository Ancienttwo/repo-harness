# 总体判断

你的方向是成立的，但需要做一个关键修正：

> 不要把这个模块设计成“让 Agent 写得更完整的需求系统”，而要把它设计成“阻止 Agent 在证据不足时扩张产品表面与实现范围的裁决系统”。

`repo-harness` 本身已经把 PRD、Sprint、任务契约、检查证据和交接信息落到仓库文件中，并通过轻量 skill facade 调用底层契约、脚本与 hooks，而不是把整个工作流写进 prompt。这非常适合增加一个位于 **Idea/Goal 与 PRD 之间** 的行为发现层。

不过，你提出的因果关系：

> Coding Agent 过度工程，是因为缺少用户旅程、优秀参照物和 hypothesis 框架

目前更适合作为这个模块的**首要待验证假设**，而不是既定事实。一个 2026 年的代码质量基准确实观察到 Agent 输出经常出现冗长和质量侵蚀；另一些研究也指出，Agent 缺少仓库历史、惯例和偏好时，更容易产生不自然的改动。但“补充产品上下文能否稳定减少产品过度设计”仍需通过你自己的历史任务集来验证。

---

# 一、模块的准确定位：BDD²

建议公开名称仍然保持简单：

```text
repo-harness-bdd
```

但在设计文档中明确它的完整含义：

```text
BDD² = Behavior-Driven Discovery
     + Behavior-Driven Development
```

这不是要重新定义传统 BDD，而是把两段工作衔接起来。

经典 BDD 通常分为：

1. Discovery：通过对话、规则和具体例子发现行为；
2. Formulation：把例子表达成清晰、可共享的行为规格；
3. Automation：将关键行为转成可执行检查。

它强调的是规则、具体例子和未决问题，而不是机械地产生大量 Gherkin。好的场景也应描述用户可观察的行为，而不是点击哪个按钮、调用哪个函数等实现细节。

而你的模块应当把 Discovery 再向产品上游延伸：

```text
Idea / Goal
    ↓
BDD²：行为发现、旅程、前后台边界、证据、假设、范围预算
    ↓ 人类 Freeze
PRD：批准后的产品契约
    ↓
Sprint：实现切片与顺序
    ↓
Task Contract
    ↓
Coding Agent
    ↓
行为证据 + Surface Drift 检查
```

它的核心产物不是“功能列表”，而是：

> **在给定用户、情境和目标下，系统最少必须暴露什么行为，以及明确不暴露什么。**

---

# 二、它与 repo-harness 现有模块的边界

`repo-harness` 已经有 `$geju`、PRD、Sprint、prior-art、negative scenarios、前端设计 brief 和 human confirmation 等能力，因此 BDD 模块不能重复创建另一套产品文档体系。现有 PRD 还明确要求非目标、负向行为场景和必要的相邻产品模式；Sprint 则从 PRD 中提取行为和场景形成可实现 backlog。

建议这样分工：

| 层级 | 回答的问题 | 不负责什么 |
|---|---|---|
| `$geju` | 我们相信什么方向？什么会证伪这个方向？ | 不展开完整用户旅程 |
| `BDD²` | 用户应发生什么行为？最小产品表面是什么？哪些是假设？ | 不安排工程任务 |
| `PRD` | 哪些产品行为已经获批并成为交付契约？ | 不重新做开放式探索 |
| `Sprint` | 如何切片、排序、验证实现？ | 不扩大产品范围 |
| Task Contract | 当前 Agent 允许改什么、必须验证什么？ | 不改变产品定义 |
| Minimal-change hooks | 实际改动是否偏离被冻结的行为与范围预算？ | 不代替测试和人工判断 |

一句话概括：

> `$geju` 决定方向，BDD² 决定行为边界，PRD 决定承诺，Sprint 决定实现顺序。

# 三、核心模型：只保留五个决策对象

产品理论可以很多，但不能把所有理论名词都暴露给 Agent。内部可以使用故事地图、服务蓝图、Example Mapping、Lean UX、Opportunity Solution Tree、Test Card 等方法；对外只保留五个需要做决定的对象。

## 1. Target Behavior：目标行为

不要从“做什么功能”开始，而从下面四件事开始：

```text
谁
在什么情境下
试图完成什么变化
成功之后会采取什么不同的行为
```

例如，不是：

```text
做一个 AI 工作流管理页面
```

而是：

```text
第一次使用产品的普通用户，
在没有理解模型、队列和 Agent 编排概念的情况下，
能够发起一项工作、看懂当前状态，
并在失败时知道下一步该做什么。
```

这会自然压制“把后台能力全部映射成前端按钮”的倾向。

---

## 2. Critical Journey：关键用户旅程

故事地图的价值不是把 backlog 摊开，而是把用户及其实际行动保持在讨论中心，防止团队围绕孤立功能争论。

但仅有用户旅程还不够。它只能告诉你用户在前台做什么，不能判断哪些后台复杂度需要暴露。因此需要同时引入服务蓝图，将每个触点拆成：

| 层 | 内容 |
|---|---|
| 用户意图 | 用户此刻真正想完成什么 |
| 用户动作 | 用户必须主动做什么 |
| 可见响应 | 用户需要看到什么才能继续 |
| 后台支持 | 系统内部为了完成结果做什么 |
| 失败与恢复 | 失败时用户如何理解、退出、重试或撤销 |

服务蓝图本来就是用于连接用户旅程触点与前台、后台人员、流程和系统依赖的工具。

一个关键规则是：

> 后台流程存在，不等于前端必须存在对应控制项。

---

## 3. Exposure Boundary：前后台暴露边界

这是整个模块最重要、也是你当前设想里最值得强化的部分。

一项能力只在以下情形进入前台：

- 用户必须明确发起它；
- 它会改变用户承诺、成本、权限或数据；
- 用户需要选择不同结果；
- 用户必须理解当前状态才能继续；
- 用户需要撤销、取消、恢复或纠错；
- 不显示会破坏信任、知情或控制感。

下面这些通常应留在后台：

- 自动重试策略；
- 缓存与预取；
- 队列和任务调度；
- 模型路由；
- fallback 顺序；
- 内部状态机；
- 日志等级；
- 索引、分片和连接池；
- Agent 间协调；
- 实现模式和技术参数。

少数高级用户确实需要的能力，采用渐进披露，而不是默认平铺。渐进披露可以把低频、复杂选项延后，但系统状态、可恢复性和用户控制不能因此消失。

建议 BDD 模块要求每个前端元素回答：

```text
这个元素帮助用户做出什么必要决定？

删除它后，用户是否仍然可以：
1. 完成主要目标；
2. 理解结果；
3. 控制高风险行为；
4. 从错误中恢复？
```

如果四项都不受影响，它默认不应存在。

---

## 4. Risk Hypothesis：风险假设

不要让 Agent直接从需求跳到方案。BDD² 应先要求识别四类风险：

- Value：用户是否真的需要；
- Usability：用户是否能理解和完成；
- Feasibility：在当前技术、成本、延迟约束下是否能做；
- Viability：是否符合业务、法律、安全和运营约束。

这是现代产品发现中常用的四风险模型。

每个高风险假设采用统一格式：

```text
我们相信：
[特定用户] 在 [情境] 中，
如果获得 [最小行为能力]，
将更可能完成 [可观察结果]。

当前证据：
[...]

最低成本测试：
[...]

观察指标：
[...]

证伪阈值：
[...]

证伪后动作：
删除 / 收缩 / 改变边界 / 暂缓
```

Opportunity Solution Tree 强调从 outcome 到 opportunity、solution 和 assumption tests；Strategyzer 的 Test Card 则要求明确假设、测试、指标和阈值。两者都适合作为内部生成方法。

重要的是：

> Hypothesis 不是“写一句看起来科学的话”，而是提前定义什么证据会让我们停止做这项功能。

---

## 5. Scope / Surface Budget：范围与产品表面预算

“Less is more”不能只是一句审美原则，必须转成可检查的约束。

建议不使用一个容易被游戏化的综合评分，而是维护一个 **Surface Delta 向量**：

```text
产品表面
Δ 顶级页面 / 路由
Δ 主要动作
Δ 新用户概念和名词
Δ 设置项
Δ 角色与权限
Δ 持久状态
Δ 通知类型

用户旅程
Δ 必经步骤
Δ 用户决策点
Δ 中断与跳转
Δ 跨角色交接

工程表面
Δ 外部依赖
Δ 新文件
Δ 新抽象
Δ 新服务
Δ 数据模型
Δ 修改超出任务契约的路径
```

每个 Behavior Brief 应给出预算，例如：

```yaml
surface_budget:
  top_level_routes: 0
  primary_actions: 1
  new_user_concepts: 1
  settings: 0
  roles: 0
  persistent_states: 1

journey_budget:
  added_required_steps: 0
  added_decisions: 1

implementation_budget:
  new_external_dependencies: 0
  new_services: 0
```

这些数字不是普适 UX 定律，而是本次工作的 appetite 和默认上限。

Shape Up 中的 appetite 本质上就是固定投入、可变范围的约束：先决定这件事值得花多少，再塑造能在该范围内成立的方案，并明确哪些内容不做。

# 四、唯一的核心产物：Behavior Brief

为了避免“反过度工程模块自己过度工程”，第一版不要创建：

```text
journey.md
hypothesis.md
scenarios.md
references.md
prototype.md
decision.md
```

建议默认只有一个人工维护文件：

```text
plans/behaviors/
└── 20260711-1430-simple-work-entry.behavior.md
```

再加一个由工具生成、不可手工维护的结构化 sidecar：

```text
.ai/harness/bdd/
└── simple-work-entry.json
```

运行时截图、原型和临时分析默认进入 ignored 路径：

```text
.ai/harness/bdd/runs/<run-id>/
```

只有被明确选中的参考证据和设计资产才进入版本控制。

## Behavior Brief 建议结构

```markdown
---
type: behavior-brief
status: shaped
owner: product
source: idea
appetite: small
created_at: 2026-07-11
---

# Outcome

## Target actor
谁是本次行为的主角。
默认只允许一个主要角色。

## Situation
他在什么情况下进入这个流程。

## Desired behavioral change
用户完成后，会做出什么可观察的不同动作。

## Current truth
当前产品、仓库和实际工作流已经如何运作。
不要把 brownfield 项目假想成 greenfield。

# Critical Journey

| Step | User intent/action | Visible response | Backstage support | Failure/recovery |
|---|---|---|---|---|

# Exposure Decisions

| Capability | Show / Hide / Progressive | Why user needs it | Cost of exposure |
|---|---|---|---|

# Surface Budget

...

# Rules, Examples, Questions

## Rule 1
...

### Example
Given ...
When ...
Then ...

### Negative example
Given ...
When ...
Then ...

### Open question
...

# Risky Assumptions

| Risk | Hypothesis | Evidence | Test | Falsifier |
|---|---|---|---|---|

# Reference Patterns

| Source | Pattern | Adopt / Adapt / Avoid | Provenance |
|---|---|---|---|

# Non-goals and No-gos

明确本次不增加的页面、控制项、角色、概念、设置和技术扩张。

# Decision

Proceed / Reshape / Defer / Kill

# PRD Export

仅包含已经冻结、可以成为产品承诺的内容。
```

## 场景应该长这样

```gherkin
Rule: 普通用户不需要理解后台编排即可开始工作

Given 用户当前只有一个主要工作目标
When 用户进入工作区
Then 系统只显示开始该工作的主要动作
And 不显示模型路由、队列、重试策略或 Agent 编排设置
```

负向行为同样重要：

```gherkin
Rule: 后台自动恢复不产生新的用户配置负担

Given 一次后台调用发生了可自动恢复的暂时失败
When 系统自动重试并最终成功
Then 用户看到最终完成状态
And 界面不新增重试次数或 fallback 顺序设置
```

只有当自动恢复失败、持续时间影响用户行动，或者可能产生额外成本时，才显示必要状态和恢复动作。

# 五、Skills 架构：一个公开入口，多个内部能力

不要给普通使用者暴露七八个命令。公开入口只有：

```text
repo-harness-bdd
```

它根据 brief 状态路由到内部 skills。`repo-harness` 当前 skill 本身就是 compatibility router，真正权威逻辑应留在 CLI、契约、脚本和 hooks 中，所以这种 facade 设计与现有架构一致。

内部建议拆成七项能力：

| Internal skill | 职责 | 主要输出 |
|---|---|---|
| `bdd-frame` | 定义用户、情境、目标行为、appetite、当前事实 | Outcome |
| `bdd-journey` | 构建关键旅程和轻量服务蓝图 | Journey |
| `bdd-boundary` | 决定前台、后台、渐进披露和 Surface Budget | Exposure Decisions |
| `bdd-evidence` | 搜集仓库、用户、分析、支持记录与参考模式 | Evidence Cards |
| `bdd-hypothesis` | 风险分类、测试、指标、证伪阈值 | Hypotheses |
| `bdd-scenario` | Example Mapping、规则、例子、问题和负向场景 | Behavior Scenarios |
| `bdd-review-export` | 检查、冻结并投影到 PRD | Frozen Brief / PRD Input |

可选的原型能力不必作为常驻核心 skill：

```text
bdd-prototype
```

它仅在可用性或信息架构风险达到阈值时触发。

# 六、工具集设计

## 1. Behavior CLI

推荐命令面：

```bash
repo-harness bdd start "<idea>"
repo-harness bdd inspect <brief>
repo-harness bdd check <brief>
repo-harness bdd review <brief>
repo-harness bdd freeze <brief>
repo-harness bdd diff <brief>
repo-harness bdd export-prd <brief>
```

真正高频的只有：

```text
start → check → freeze → export-prd
```

`freeze` 和 `export-prd` 是硬校验点；探索过程保持 advisory，避免 Agent 在还不确定时为了“通过 schema”伪造答案。

---

## 2. Journey / Scenario Compiler

它不是自动编造完整旅程，而是做结构转换：

```text
Journey step
  → behavior rule
  → concrete example
  → negative example
  → unresolved question
  → possible acceptance evidence
```

它还负责发现：

- 一个场景是否描述了实现步骤而不是行为；
- 是否只有 happy path；
- 是否遗漏取消、撤销、超时、空状态、权限变化和恢复；
- 是否偷偷引入新的角色、设置或概念；
- 同一个用户目标是否被拆成过多页面。

---

## 3. Surface Boundary Linter

这是 BDD 模块最有差异化价值的工具。

它可以同时检查 Behavior Brief、PRD、Sprint 和代码 diff：

```text
Brief 允许：
- 0 个新顶级路由
- 1 个新主要动作
- 0 个新设置
- 1 个新用户概念

实际 diff 发现：
- 2 个新路由
- 4 个新按钮
- 1 个 Advanced Settings
- 3 个新用户可见名词
```

输出不应只是“失败”，而要按原因分类：

```text
UNBUDGETED_SURFACE
UNEXPLAINED_USER_CONCEPT
BACKSTAGE_LEAK
JOURNEY_STEP_GROWTH
NEW_ROLE_WITHOUT_BEHAVIOR
SETTINGS_AS_IMPLEMENTATION_ESCAPE
OUT_OF_CONTRACT_CHANGE
```

特别值得加入一条启发式：

> 当 UI 文案直接暴露技术实现名词，而 Behavior Brief 中没有对应的用户决策需求时，标记为 `BACKSTAGE_LEAK`。

---

## 4. Chrome Reference Collector

Chrome 插件不应定位成“批量抓竞品”，而应定位成：

> 经用户明确操作采集的、带来源与设计判断的 Pattern Card。

单次采集流程：

```text
用户主动点击插件
    ↓
选择当前视口或指定元素
    ↓
本地遮盖个人信息、账号、密钥和客户数据
    ↓
保存截图及元数据
    ↓
要求填写：Adopt / Adapt / Avoid
    ↓
关联到具体 journey step 或 exposure decision
```

每张卡至少包含：

```yaml
source_url:
page_title:
captured_at:
viewport:
element_description:
journey_step:
observed_pattern:
why_it_works:
adopt_adapt_avoid:
privacy_review:
content_hash:
```

权限上，应优先采用用户操作时请求的 optional permissions 或 active-tab 式访问，而不是安装即获得大范围常驻权限；同时需要明确披露数据用途和存储策略。Chrome 官方扩展文档也建议在运行时请求可选权限，并对用户数据的收集、使用和披露施加约束。

更重要的是：

> 截图证明“某个成熟产品采用了某种模式”，不证明“我们的用户需要这个功能”。

所以 Pattern Card 的结论只能是：

```text
Adopt：相同用户意图和约束下可直接采用
Adapt：意图类似，但复杂度、角色或频率不同
Avoid：会引入不必要概念、设置或导航
```

不能只是“竞品有，所以我们也要有”。

---

## 5. Prototype Generator

GPT Image 适合快速产生视觉方向和修改原型，目前官方图像模型支持从文本和图像输入进行生成与编辑；官方提示指南也强调清晰约束和小步迭代。

但必须明确：

```text
原型是刺激材料，不是用户证据。
视觉完整度不是行为正确性的证据。
AI 生成的用户反馈不是真实用户验证。
```

生成器不应只给一个“漂亮方案”，而应围绕同一行为生成三种有明确假设差异的低保真方案：

1. **Minimum Surface**：只保留主路径和恢复动作；
2. **Progressive Disclosure**：高级能力延后；
3. **Expert Path**：给高频专家更快路径，但不污染默认界面。

每个方案必须附带：

```text
它减少了哪些用户决策？
它新增了什么概念？
哪些后台状态被隐藏？
哪些风险因此增加？
通过什么观察能淘汰它？
```

只有行为边界已经基本稳定，且存在明显可用性风险时才调用原型生成，避免团队过早被视觉细节锚定。

---

## 6. Evidence / Provenance Store

证据需要按可信度分层：

### A 级：直接事实

- 当前产品真实行为；
- 仓库现有代码和契约；
- 真实分析数据；
- 用户访谈；
- 支持请求；
- 生产事故；
- 销售或运营记录。

### B 级：观察性参照

- 成熟产品的可观察模式；
- 官方产品文档；
- 真实工作流程观察；
- 专家评审。

### C 级：实验性证据

- 用户操作原型的观察；
- 可用性测试；
- concierge 或 wizard-of-oz 实验；
- 小范围发布。

### D 级：合成材料

- ImageGen 原型；
- AI 生成 persona；
- AI 猜测的用户反馈；
- 没有来源的“行业最佳实践”。

D 级只能产生假设，不能关闭假设。

每条证据都应有：

```text
来源
采集时间
对应假设
新鲜度
是否一手证据
是否存在隐私限制
它支持什么
它不能证明什么
```

---

## 7. MCP Planner Tools

`repo-harness` 已有可选 planner MCP，支持 ChatGPT 读取仓库产物，并以受保护的 hash 写入方式创建 PRD、Sprint 和 goal；默认不允许随意写源代码或执行 shell。BDD 可以沿用同样的安全边界。

建议增加：

```text
read_behavior_context
write_behavior_brief_from_idea
append_behavior_evidence
update_behavior_decision
check_behavior_brief
freeze_behavior_brief
write_prd_from_behavior
```

其中：

- `append_behavior_evidence` 不能覆盖人工决策；
- `freeze_behavior_brief` 必须带预期 hash；
- `write_prd_from_behavior` 只导出 frozen 内容；
- Agent 不允许在导出时重新扩展范围。

# 七、状态机与审查门

建议状态机：

```text
raw
  ↓
framed
  ↓
evidenced
  ↓
shaped
  ↓
reviewed
  ↓ human action
frozen
  ↓
exported
```

旁路状态：

```text
reshape
deferred
killed
```

## Freeze 的硬阻塞条件

出现下面任一情况，不允许冻结：

1. 没有明确的目标用户行为；
2. 没有关键旅程起点和终点；
3. 把当前产品事实和设想混在一起；
4. 没有前台/后台暴露决策；
5. 没有 Non-goals 或 Surface Budget；
6. 高风险假设没有测试或证伪阈值；
7. 把竞品截图当作用户需求证据；
8. 把 AI 原型或合成 persona 当作真实验证；
9. 只写 happy path，没有失败恢复；
10. 引入了新角色、权限、设置或页面，却没有用户行为理由；
11. 存在高后果动作，但没有确认、撤销或恢复策略；
12. Critical questions 仍然会改变主旅程或产品表面。

## 人类审批应该审批什么

不要让人类只点“PRD 看起来不错”。

冻结界面只要求人类确认五件事：

```text
1. 目标行为对吗？
2. 关键旅程完整吗？
3. 暴露给用户的东西是否必要？
4. 最大风险和证伪条件对吗？
5. 范围预算是否合理？
```

这正好对应前面的五个决策对象。

# 八、什么时候必须运行 BDD，什么时候跳过

并非所有工作都需要产品发现。风险较小、行为已经定义清楚的工作，可以直接进入 PRD、Sprint 或 Task Contract；风险发现本身也应与任务风险匹配，而不是成为新的官僚流程。

## 默认必须触发

- 新的用户可见能力；
- 新的顶级页面或导航；
- 新角色或权限；
- 支付、计费和价格；
- 删除、覆盖、发布等高后果行为；
- AI 自动行动或自主 Agent 行为；
- 外部系统集成；
- 改变多个模块之间的用户旅程；
- 新设置中心或高级配置；
- 改变用户心智模型的术语或状态。

## 建议触发

- 显著修改现有流程；
- 新增长期持久状态；
- 同一目标存在多个相互竞争的交互方案；
- 用户频率、专业程度或使用情境不明确。

## 默认跳过

- 恢复既定行为的 bugfix；
- 内部重构；
- 依赖升级；
- 文档修订；
- 测试补齐；
- 已有产品契约下的微小视觉修复；
- 紧急安全修复。

跳过也应该留下机器可读原因：

```yaml
bdd:
  applicability: skipped
  reason: restore_existing_defined_behavior
  source_contract: plans/prds/...
```

# 九、与 minimal-change hooks 的结合

现有 minimal-change hooks 已经采用 opt-in、advisory、fail-open 的模式，用于记录新依赖、新文件、新抽象和受保护关注点，并明确不代替计划、测试和人工审查。BDD 的第一版应沿用这一哲学。

建议接入方式：

## SessionStart

注入被冻结 brief 的压缩摘要：

```text
Target behavior
Critical journey
Surface budget
No-gos
Protected concerns
```

不把完整研究资料全部塞进上下文。

## Prompt / Pre-plan

当任务属于必须运行 BDD 的类别但没有 frozen brief：

```text
advisory:
  BDD_REQUIRED_FOR_NEW_USER_SURFACE
```

在早期只警告，不拦截。

## Post-edit

计算实际 Surface Delta：

```text
新增路由
新增界面文件
新增设置
新增用户可见名词
新增依赖
新增抽象
修改契约外路径
```

与 frozen budget 比较。

## Stop

写出 drift summary：

```json
{
  "behavior_contract": "...",
  "budget_status": "exceeded",
  "observed": {
    "top_level_routes": 1,
    "settings": 2,
    "new_user_concepts": 4
  },
  "violations": [
    "BACKSTAGE_LEAK",
    "UNBUDGETED_SURFACE"
  ]
}
```

最开始不要自动失败。只有当历史评估证明规则稳定、误报可控后，再把少数规则升级为 gate。

# 十、你原始设想中还缺少的部分

## 1. Brownfield Truth

Agent 很容易把已有产品当作全新产品重新设计。

BDD 开始前必须读取：

- 当前页面和导航；
- 已有用户术语；
- 已有角色和权限；
- 当前数据与状态；
- 已有错误恢复方式；
- 已经存在但未文档化的行为。

否则即使有“用户旅程”，也可能生成一条理想化但与真实系统不兼容的旅程。

---

## 2. 恢复路径，而不只是主路径

“少”不能意味着隐藏必要状态。

必须覆盖：

```text
empty
loading
partial
timeout
cancel
undo
retry
offline
permission denied
conflict
stale data
irreversible action
```

尤其需要区分：

- **实现细节**：不应暴露；
- **影响用户决策的状态**：必须暴露；
- **影响信任、成本和可恢复性的事实**：必须暴露。

---

## 3. 反向风险：防止“少”变成欠工程

模块不能单向惩罚文件、抽象和错误处理，否则 Agent 会学会删掉必要的安全机制。

以下内容应列为 protected concerns，不计入普通的过度工程惩罚：

- 身份认证和权限校验；
- 输入验证；
- 数据完整性；
- 隐私和安全；
- 可访问性；
- 审计和合规；
- 事务和并发正确性；
- 错误恢复；
- 回滚与迁移；
- 测试；
- 明确要求的性能约束。

`less is more` 应用于**不必要的用户和工程表面**，而不是削弱正确性。

---

## 4. 多角色管理

很多复杂度来自 Agent 默认替所有潜在角色设计完整界面。

建议规则：

```text
一个 brief 默认只有一个 primary actor。
secondary actor 只有在价值流无法完成时才进入主 brief。
admin、support、operator 的后台需求默认进入 separate behavior slice。
```

这可以防止一个普通用户需求最后长成“用户端 + 管理后台 + 审计中心 + 高级配置”。

---

## 5. 迁移和兼容性

新行为不仅要描述目标状态，还需要回答：

- 现有用户看到什么；
- 旧数据如何处理；
- 旧入口是否保留；
- 名词是否变化；
- 默认值是否改变；
- 是否需要渐进迁移；
- 如何撤回。

否则 Agent 很容易通过“重做整个系统”获得局部简洁。

---

## 6. 参照物的锚定风险

优秀产品截图非常有价值，但也容易造成三个问题：

- 因为对方有某功能而复制；
- 因为视觉成熟而误以为行为正确；
- 忽视双方用户频率、商业模式、权限和技术约束不同。

所以截图采集必须绑定一个问题：

```text
这张参考图帮助我们解决哪个 journey step、
exposure decision 或 recovery problem？
```

没有绑定问题的截图不进入正式证据库。

---

## 7. Discovery 的停止规则

没有停止规则的产品发现同样会过度工程。

建议在开始前设置 appetite：

```text
tiny: 15–30 分钟，只做 frame + boundary
small: 一次结构化 review，不做用户原型
medium: 允许参考采集和一个原型实验
large: 需要真实用户或生产证据
```

这里表达的是投入等级，不是要求系统计时或后台异步工作。

停止条件是：

```text
关键旅程已清楚
高风险假设有处理方式
前后台边界已决定
Surface Budget 已批准
剩余问题不会改变主行为
```

而不是“所有问题都回答完”。

# 十三、如何验证它真的减少了过度工程

这是不可省略的一环，否则该模块本身会成为一套未经验证的产品理论。

## 建立历史任务集

选择 20–50 个过去真实任务，最好包含：

- Agent 增加过多页面；
- 增加无用设置；
- 把后台概念暴露给用户；
- 创建多余抽象；
- 修改范围溢出；
- 看似完整但主旅程反而更差；
- 同时包含一些确实需要复杂实现的任务。

对每个任务保存：

```text
原始需求
仓库初始状态
历史产出
人工指出的问题
理想最小行为
不可删除的安全/恢复要求
```

## A/B 对照

保持模型、工具和任务相同：

```text
A：现有 repo-harness PRD → Sprint 流程
B：BDD² → PRD → Sprint 流程
```

由不知道产出来源的产品、设计和工程评审者盲评。

## 主要指标

### 产品结果

- 主旅程是否正确；
- 用户是否能完成目标；
- 产品表面是否必要；
- 是否引入无依据的新概念；
- 是否符合 Non-goals；
- 是否满足恢复和信任要求。

### 客观 Surface Delta

- 新页面和路由；
- 新主要动作；
- 新设置；
- 新角色；
- 新用户术语；
- 新依赖；
- 新文件与抽象；
- 契约外修改路径。

### 工程结果

- 验收通过率；
- 回归；
- 测试完整性；
- 安全与数据正确性；
- 人工返工量；
- 后续删除的代码比例。

### 反向指标

必须专门监测：

- 是否遗漏错误处理；
- 是否遗漏权限校验；
- 是否牺牲可访问性；
- 是否隐藏了用户必须知道的状态；
- 是否因强行“少”而产生脆弱耦合；
- 是否把复杂度推迟到未来而没有记录。

只有在 BDD 组稳定降低产品表面和范围偏移，同时没有降低验收、安全和恢复质量时，才证明你的核心假设成立。

# 十四、实施路线：不要一开始做 Chrome 插件和 ImageGen 全家桶

## Phase 0：先做评估基础

- 整理历史过度工程案例；
- 定义 Surface Delta；
- 写盲评 rubric；
- 建立 10–20 个 golden behavior briefs。

## Phase 1：最小 BDD 核心

只实现：

```text
一个公开 skill
一个 Behavior Brief
一个 validator
一个 human freeze
一个 PRD exporter
```

包含四项内部能力：

```text
frame
journey
boundary
hypothesis
```

这是最值得先验证的闭环。

## Phase 2：证据与参照物

增加：

- Pattern Card；
- Chrome 显式采集；
- 来源、时间、隐私和 adopt/adapt/avoid；
- 参考模式与 journey step 绑定。

## Phase 3：风险触发的原型

增加：

- Minimum Surface；
- Progressive Disclosure；
- Expert Path；
- 原型观察记录；
- synthetic evidence 标签。

## Phase 4：实现漂移检测

增加：

- Surface Delta；
- SessionStart 行为摘要；
- post-edit drift；
- Stop summary；
- 初期 advisory。

## Phase 5：基于评估结果决定是否硬门控

只有误报足够低时，才把下面几项变成 hard gate：

```text
没有 frozen brief 却新增顶级用户表面
超出 Surface Budget
新增角色或权限但没有行为依据
明显的后台概念泄露
高后果动作没有恢复策略
```

# 最终框架

整个模块可以压缩成一句可执行定义：

> `repo-harness-bdd` 在 PRD 之前，通过目标行为、关键旅程、前后台边界、风险假设和产品表面预算，将模糊想法塑造成最小且可证伪的行为契约；在实现过程中，再用该契约检测 Agent 是否增加了未经批准的用户或工程复杂度。

第一版最重要的不是截图采集，也不是 ImageGen，而是：

```text
Behavior Brief
+ Exposure Boundary
+ Surface Budget
+ Human Freeze
+ PRD Export
+ Surface Drift Eval
```

先证明这六件事能够在真实历史任务上减少无价值页面、设置、概念和抽象，再扩展参考采集与原型工具。这样，这个“反过度工程模块”本身才不会成为一次新的过度工程。
